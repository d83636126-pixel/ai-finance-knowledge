# 突发事件：如何处理 & 如何抓取

## 1. 两类事件，一套分析流程

| 类型 | 例子 | 特点 | 抓取方式 |
|------|------|------|----------|
| **计划内** | FOMC、非农、CPI、已知选举 | 时间可预期 | 日历 + `MAJOR_EVENTS` |
| **突发** | 以伊冲突升级、SVB 倒闭、关税突发、LUNA 崩盘 | 时间不可预期 | 新闻摄入 + 市场症状 + 手动录入 |

**关键原则**：突发与计划内，在分析层走**同一条路**：

```text
事件 instance（具体一次发生）
  → 映射 graph event 节点（或最近邻）
  → 事前/即时路径报告
  → 验证指标 + evidence
  → 事后对比 + 图谱校准
```

区别只在 **「事件从哪来」**，不在分析引擎。

## 2. 现有代码能力 vs 缺口

### 已有

| 能力 | 位置 | 说明 |
|------|------|------|
| 图谱 event 节点 | `propagation_graph.json` | conflict / sanction / energy / tail_risk 等 subtype |
| 日历 overlay | `PropagationEngine.getCalendarOverlay()` | 未来 14 天 `MAJOR_EVENTS` |
| 事前/事后报告 | `scenario_reasoner.js` | `generatePreReport` / `generatePostReport` / `compareReports` |
| 关键词库 | `config.js` | `NEWS_KEYWORDS_BY_CATEGORY`、`BREAKING_KEYWORDS` |
| RSS 拉取 | `data.js` `fetchRSS()` | 已实现，**未接入扫描** |
| 市场症状信号 | `signals.js` | VIX、DXY z-score、油价 shock、BTC 动量 |
| 历史重大事件 | `MAJOR_EVENTS` | 2020–2026 手工清单，用于回测 |

### 缺口（突发相关）

1. **无新闻扫描器**：`runFullScan()` 只跑 crypto/onchain/derivatives/macro(VIX)，不扫 RSS
2. **关键词未使用**：`BREAKING_KEYWORDS` 定义了但没有匹配逻辑
3. **新闻 → 图谱无映射**：抓到标题后不知道对应哪个 `event` 节点
4. **突发 instance 无标准模型**：图谱只有「事件类型节点」，没有「2026-04-13 伊朗袭击以色列」这种 instance
5. **症状 vs 原因未桥接**：BTC 暴跌被扫到，但系统不会问「是不是地缘/监管/流动性？」

## 3. 突发事件处理流程（目标设计）

### 阶段 1：发现（Detection）

三条并行管道，**不推送，只进「事件 inbox」**：

```text
管道 A · 新闻/RSS
  Reuters / Bloomberg / 财联社 / Fed 官网 RSS
  → 关键词命中 (BREAKING_KEYWORDS)
  → 可选 LLM 分类 (地缘/宏观/监管/ crypto)
  → 写入 event_inbox

管道 B · 市场症状
  BTC/Nasdaq/VIX/DXY/Oil 超阈值
  → 多维度同时异动 → 提高优先级
  → 写入 event_inbox（标记为 symptom_trigger）

管道 C · 人工
  用户看到新闻 → 「快速录入突发」
  → 写入 event_inbox
```

### 阶段 2：归类（Classification）

对 inbox 每条记录：

```text
1. 尝试匹配 propagation_graph 已有 event 节点
   例：「伊朗导弹袭击」→ geoconflict_escalation / red_sea_shipping_disruption

2. 匹配置信度低 → 展示 Top-3 候选 + 让用户确认

3. 完全新型 → 挂到 generic tail_risk 或 macro_scenario，并标记「待补图谱」
```

建议维护一张 **Event Taxonomy Map**（可放 JSON）：

```text
关键词/模式 → graph_node_id → line → 默认验证指标列表
```

### 阶段 3：分析（Reasoning）— 即时「伪事前」

突发事件没有真正的「事前」，但有 **「发生后 0–2 小时」** 的快速分析窗口：

```text
确认 graph event
  → findPaths() / getPathReport()
  → 核心 target 验证仪表
  → 列出「接下来 24h 应盯的 verifier」
  → 保存为 sudden_pre_report（实为 instant analysis）
```

### 阶段 4：验证（Verification）— 事后对比

与计划内相同，T+1 / T+24：

```text
刷新 market_snapshot
  → evidence 状态
  → 与 instant analysis 的预期方向对比
  → 写入 post_report + compareReports
  → 可选：calibrator 反馈边 weight
```

## 4. 如何抓取「特殊事件」（分层次）

### 层次 1：规则抓取（P1 先做，成本低）

- RSS 源列表（config 可配置）
- `BREAKING_KEYWORDS` + `NEWS_KEYWORDS_BY_CATEGORY` 分层匹配
- 去重：同标题 6h 内不重复
- 输出：`{ title, url, published_at, matched_keywords, suggested_graph_nodes[] }`

### 层次 2：图谱映射（P1 同步做）

- 维护 `event_taxonomy.json`：关键词 → graph event id
- 图谱已有 subtype 可直接用：`conflict`, `sanction`, `energy`, `tail_risk`, `enforcement`…
- 映射失败 → inbox 标记 `needs_review`

### 层次 3：市场交叉验证（P1.5）

- 新闻命中 + VIX/DXY/BTC 同时异动 → 提升 `confidence`
- 仅新闻、市场无反应 → 标记 `narrative_only`，路径权重下调
- 仅市场异动、无新闻 → `symptom_trigger`，推荐用户选 narrative

### 层次 4：LLM 辅助（P2，非必须）

- 输入：标题 + 摘要 + 已匹配关键词
- 输出：`category`, `graph_node_id`, `direction_hint`, `affected_targets[]`
- **不自动改图谱**，只建议，人确认后写入

### 层次 5：人工兜底（始终保留）

- 「快速录入突发」表单：标题、时间、选 graph event、备注
- 复用 `ScenarioReasoner.saveEvent()`，增加字段 `source: sudden | scheduled | manual`

## 5. 建议的 RSS / 数据源（第一版）

| 类型 | 来源 | 用途 |
|------|------|------|
| 宏观 | Fed 新闻 RSS、BLS、Treasury | 政策/数据突发 |
| 地缘 | Reuters World、AP Top News | 冲突/制裁 |
| 市场 | 财经媒体 RSS（需 CORS 代理） | 通用突发 |
| 官方 | 各国央行公告页 | 意外降息等 |

BLS 非农已有 `scripts/fetch_bls_*.js`，属**计划内**；突发抓取是**另一管道**。

## 6. 与推送的关系

按最终用途决策：

- 突发进 **event inbox**，在 propagation 页展示「待分析突发」
- **不推微信**，直到 P3
- Dashboard 可只读展示 inbox 计数（可选，P2）

## 7. P1 实施顺序（突发专项）

1. 定义 `event_inbox.json` schema + `event_taxonomy.json`
2. 实现 `EventIngestor.scanRSS()`（接现有 `fetchRSS` + `BREAKING_KEYWORDS`）
3. propagation 页增加「事件 inbox」面板
4. 一键：inbox item → 映射 graph event → path report
5. 症状触发：复用 `SignalDataProvider`，多信号聚合时写入 inbox
6. 与 `ScenarioReasoner` 打通 sudden pre/post

## 8. 用户确认（2026-07-10）

| 项 | 决定 |
|----|------|
| RSS 源 | **中英文都要** |
| 分析触发 | **先手动**（inbox 展示 → 用户点「分析」再跑路径） |
| LLM | **要接**，见下文「LLM 策略」 |

## 9. LLM 策略：不是不接，而是分角色接入

### 为什么不建议 P1「全自动 LLM 驱动」

第一版若让 LLM 独立完成「抓新闻 → 定事件 → 改图谱 → 出结论」，常见问题是：

1. **误分类成本高**：突发标题歧义大（例：「Iran talks」可能是缓和或升级），自动映射到错误 graph event 会带偏整条传导链。
2. **图谱是决策资产**：边权重、event 节点是人工治理过的；LLM 自动写入图谱会造成治理回退。
3. **可验证性**：事前/事后对比要求每个结论能追溯到「哪条边、哪个 verifier」；纯 LLM 摘要难以审计。
4. **成本与延迟**：中英文 RSS 高频扫描 + 每条都调 LLM，P1 阶段性价比低；关键词已能过滤 80% 噪音。
5. **项目已有 LLM 用法**：`Evaluator` 是对**已确认信号**做评价，不是对**未确认新闻**做自动定性与推送——应延续「人定框架、机做辅助」。

### 推荐接法：LLM 做「归类助手」，不是「自动决策者」

```text
RSS + 关键词（规则层，廉价、可审计）
  → inbox 条目
  → 用户点「分析」或「LLM 归类」
  → LLM 输出：category / suggested_graph_nodes[] / direction_hint / 一句话摘要
  → 用户确认 graph event（手动兜底始终保留）
  → PropagationEngine 跑路径（确定性引擎，非 LLM 幻觉）
  → evidence 层用真实数据验证（FRED/DXY/MOVE）
```

这样 LLM 负责 **理解新闻语义**，图谱引擎负责 **传导推理**，人负责 **最终确认**——与「先手动」不矛盾。

### P1 LLM 最小接入

- 复用现有 `Evaluator` / `DataAPI.callDeepSeek` 基础设施
- 新增 `EventClassifier.classify(headline, snippet)`：
  - 输入：标题 + 摘要 + 已命中关键词
  - 输出：JSON `{ category, graph_node_candidates[], confidence, summary_zh }`
  - **不写图谱、不自动跑路径**，只填充 inbox 的「建议映射」
- 用户在 inbox 一键「采用 LLM 建议」或手动改选

### P2 再考虑

- 新闻 + 市场症状 + LLM 三者一致时，提高 inbox 排序（仍不自动推送）
- 批量事后对比时，LLM 生成「叙事 vs 实际」文字摘要
