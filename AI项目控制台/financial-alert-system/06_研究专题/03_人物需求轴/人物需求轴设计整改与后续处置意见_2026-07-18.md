# 人物需求轴：设计整改与后续处置意见（2026-07-18）

> [!danger] 当前决策
> 人物需求轴的研究问题有价值，但原方案在当前项目阶段仍然过大，也过早把“研究假设”写成了“产品脊梁”。
>
> 本专题保留为Obsidian研究假设，**停止产品化、停止代码开发、停止UI和自动采集扩展**。在NFP核心闭环没有完成研究验证之前，不得重新进入financial-alert-system主开发线。

## 一、对原设计的真实判断

原稿有两个值得保留的核心：

1. 不做人物百科，而是研究人物在制度和利益约束下可能采取的动作；
2. 每个推断必须有证据、时间边界和可证伪条件。

但原稿仍有明显问题：

| 问题 | 原设计表现 | 风险 |
|---|---|---|
| 范围仍然过大 | 首批22–28人，后续接Inbox、图谱、NFP Case | 再次形成“大系统先行、真实价值未验”的路径 |
| 单轴解释过强 | 多数行动者围绕`TrumpDemand(t)`解释 | 容易把复杂制度行为压成个人忠诚叙事 |
| 主观字段过多 | Want、Lack、Alignment等容易凭解释填写 | 读起来合理，但无法稳定复核和结算 |
| 产品化过早 | 被称为愿景核心脊梁和下一核心 | 在NFP实际功能尚未证明时继续横向扩张 |
| 维护成本被低估 | 预计每周2–4小时、名单持续扩充 | 实际会演变成新闻整理和人物档案维护 |
| 缺少简单基线 | 没说明比“官方议程+发言时间线”多提供了什么 | 无法证明复杂模型值得存在 |

因此，不能因为概念上“很有洞察力”就把它继续建设为产品功能。

## 二、定位调整

### 原定位

```text
人物—发言—立场主库
    ↓
政策脉络OS
    ↓
传导图 / NFP / 微观层 / 交易辅助
```

### 调整后定位

```text
公开证据
    ↓
政策行动者约束台账
    ↓
少量事前动作假设
    ↓
事件后结算
    ↓
判断这种研究方法是否有增量
```

调整后的对象不是“人物内心需求”，而是：

> **公开证据能够支持的行动约束、可用工具、预期动作和证伪结果。**

建议在后续实际使用中，将“人物需求轴”工作名称逐步改为：

> **政策行动者约束台账（Policy Actor Constraint Ledger）**

原名称可以保留为研究入口，但不应把心理推断当成事实字段。

## 三、从单一TrumpDemand改为多约束模型

`TrumpDemand(t)`可以保留，但只能是一个外部约束，不再作为解释所有人物行为的唯一根节点。

每个行动者至少分成四类约束：

| 约束轴 | 含义 | 允许证据 |
|---|---|---|
| ExecutiveDemand | 行政权当前公开表达的目标与优先级 | 正式讲话、行政文件、预算和公开任命要求 |
| InstitutionalMandate | 该机构、职位和法律授权允许或要求其做什么 | 法律、章程、职责、正式程序 |
| PersonalIncentive | 可观察的职业、政治和声誉激励 | 任命、公开竞选、披露、历史公开行为；不得推断内心忠诚 |
| ExternalConstraint | 市场、国会、法院、利益冲突、数据和联盟约束 | 可核验证据和现实否决条件 |

最终的`ExpectedMove`必须来自四类约束的组合，而不是先认定“服务特朗普”，再寻找支持证据。

## 四、最小研究版本

原稿中的22–28人首批规模取消。若未来重新启动，只允许做一个手工研究试验：

| 项 | 新上限 |
|---|---:|
| 行动者 | 4–6人 |
| 研究议题 | 不超过3个 |
| 观察周期 | 90天 |
| 数据方式 | 全人工、公开来源 |
| 自动化 | 0 |
| UI | 0 |
| 图谱扩展 | 0 |
| 交易信号接入 | 0 |

首轮人物选择不按“知名度”或“完整内阁”决定，只保留：

1. 一个公开需求源；
2. 两到三个握有实际工具的执行节点；
3. 一个可能偏离行政需求的制度或市场对照节点。

对照节点是必须项，否则模型会天然把所有行为都解释为对齐，失去证伪能力。

## 五、字段整改

### 删除或降级

- `Want`：不得以心理描述直接入事实层；只能写“可观察激励”。
- `Lack`：只有权限、程序、票数、数据、资金等可核验缺口才能填写。
- `Alignment`：不得只填高/中/低；必须绑定议题、时间和观察证据。
- “内心忠诚”“真实意图”“一定会做”：禁止入库。

### 保留并加强

| 字段 | 要求 |
|---|---|
| actor_id | 稳定唯一标识 |
| role_as_of | 当时的正式角色和来源 |
| topic | 本条假设对应的具体议题 |
| as_of | 信息截止时间 |
| evidence | 原始来源、发布日期、摘录位置、信源级别 |
| fact | 仅记录来源直接支持的内容 |
| inference | 与事实严格分栏 |
| constraints | 四类约束分别填写，可为空 |
| expected_move | 可观察、可判定的动作 |
| horizon | 明确截止日期或事件窗口 |
| probability | 事前概率，不用“可能、大概”等模糊词 |
| falsifier | 出现什么证据会降低或推翻判断 |
| baseline | 不使用人物需求模型时的简单判断 |
| outcome | 截止期后的实际结果 |
| settlement | 命中/失败/无法判定及原因 |

## 六、强制研究流程

```mermaid
flowchart LR
    E["公开原始证据"] --> F["事实层"]
    F --> C["四类约束"]
    C --> H["动作假设+概率+期限"]
    H --> Z["冻结as_of与哈希"]
    Z --> O["观察实际动作"]
    O --> S["结算并记录失败原因"]
    S --> B["与简单基线比较"]
```

规则：

1. 先确定具体议题和结算日期，再收集人物材料；
2. 事实、推断、情景必须分栏；
3. `ExpectedMove`必须在截止日前冻结，事后不得改写；
4. 没发生、发生相反动作或无法判定都必须结算；
5. 不允许只保留“看起来说对了”的案例；
6. 必须与简单基线比较，例如官方议程、机构常规职责或发言时间线；
7. LLM只允许做摘录和草稿整理，不能自动决定人物意图或研究结论。

## 七、与当前项目的边界

### 当前允许

- 保存原稿和本整改意见；
- 人工记录偶发的重要人物证据；
- 作为未来研究候选，不计入产品完成度。

### 当前禁止

- 新建人物数据库代码；
- 开发人物库页面、搜索、图谱或自动抓取；
- 扩展到沃什全组、完整内阁或大规模名单；
- 将人物推断直接写入NFP研究结果；
- 将人物需求轴作为交易信号或策略权重；
- 因为文档和模板齐全就标记“人物层完成”。

人物需求轴不得与当前NFP研究样本争夺开发时间。NFP没有形成可信闭环时，人物库做得越完整，项目越容易重新回到“概念很多、实际功能没有”的状态。

## 八、是否独立成项目

当前不拆成代码项目，只保留为Obsidian研究专题。

只有满足以下条件，才允许评审是否独立为`policy-actor-evidence`项目：

- 完成至少10条事前冻结的动作假设；
- 所有假设100%按期结算；
- 原始来源和`as_of`完整率达到90%以上；
- 与“官方议程+机构常规行为”简单基线完成比较；
- 能证明减少遗漏、提高判断质量或节省时间；
- 人工维护成本可接受；
- 不依赖financial-alert-system的全局状态和UI。

若只有资料整理价值，没有可复核的预测增量，则最终形态应是：

> **政策行动者证据索引**，而不是预测系统。

## 九、重新启动条件

人物需求轴的产品化继续冻结。满足以下条件后，用户才能决定是否启动90天小试验：

1. NFP核心证据闭环已经稳定运行；
2. 项目不再处于同时补工程、研究和发布阻断的状态；
3. 已确定一个具体政策议题，而不是“研究所有人物”；
4. 已明确4–6名行动者和一个对照节点；
5. 已接受“试验可能证明这套方法没有增量”。

即使重新启动，也先在Obsidian手工运行，不进入代码仓库。

## 十、当前处置结论

| 项 | 处置 |
|---|---|
| 原设计稿 | 保留，标记为建模假设来源 |
| TrumpDemand | 保留为一条外部约束，不再作为唯一解释轴 |
| 人物数量 | 从22–28人缩减为未来试验最多4–6人 |
| 人物库代码 | 冻结 |
| Inbox/RSS自动化 | 取消当前排期 |
| 图谱/NFP/微观接入 | 禁止当前接入 |
| LLM人物模拟 | 不作为目标 |
| 当前产品状态 | 不计入功能完成度 |
| 当前研究状态 | `frozen_research` |

一句话：

> 人物需求轴可以保留为有潜力的研究方法，但不能再作为当前项目继续扩建的理由；先用少量、可冻结、可结算的动作假设证明它比简单时间线更有价值，再谈项目化。
