# 微观层整体重构：Cursor 逐步执行清单

## 0. 本计划的用途

本文件是微观层下一轮调整的**主执行单**。Cursor 必须按编号逐项完成，每次只执行一个切片；当前切片通过验收并留下证据后，才能进入下一项。

最终目标不是再增加一组指标卡，而是把微观层升级为一套可以回答以下问题的研究工作台：

1. 当前宏观情景通过什么机制传导？
2. 哪些微观证据支持、反驳或尚不足以验证这条路径？
3. 路径的哪一段已经确认，哪一段过期、冲突或缺失？
4. 当前结论的可信度由覆盖率、新鲜度、来源可信、证据一致性和历史校准中的哪些部分构成？
5. 24 / 48 / 72 小时后，路径是否被结果验证，是否需要修正？
6. 研究是否达到“可供人工决策参考”的准备度，而不是是否可以自动下单？

目标闭环：

> 宏观情景 → 传导机制 → 微观因子 → 市场状态 → 资产确认 → 研究结论 → 人工决策准备度 → 24/48/72h 结果 → 校准与路径修正

## 1. 不可突破的边界

- 项目继续以**宏观推导 + 微观分析**为主，不向自动交易延伸。
- 不接入交易所私钥、交易密钥、下单接口、订单路由、自动仓位调整或交易授权。
- 所有研究输出必须保持 `advisory-only`，并明确 `executable: false`、`human_review_required: true`。
- 数据不足、过期、冲突或校准不足时，允许并鼓励输出 `abstain` / `insufficient`。
- 不为了产生方向结论而降低校准门槛。
- 合成样本只能用于开发测试，不得进入正式校准结论。
- 不自动修改正式校准簿，不中断 P3 只观察边界。
- Harness、离线测试和浏览器测试不能替代 24h、72h 与 28 天真实观察。

如果任何任务需要突破以上边界，Cursor 必须停止并请求人工确认，不得自行推断授权。

## 3. Cursor 执行纪律

### 3.1 每次只做一个切片

每次执行开始前，先输出：

- 当前任务编号；
- 本次目标；
- 预计修改文件；
- 会复用的现有模块；
- 本次明确不修改的内容；
- 验收命令。

每次执行结束后，必须输出：

- 实际修改文件；
- 测试与验收结果；
- 新增产物路径；
- 未解决问题；
- 当前 `git status --short`；
- 是否建议进入下一切片。

### 3.2 Git 规则

- 禁止执行 `git reset --hard`、`git checkout --`、`git clean`。
- 禁止自动 push。
- 禁止把 `_workbench_check_logs/`、临时报告、令牌或运行缓存混入提交。
- 不使用 `git add .`；如需提交，只按当前切片文件范围暂存，并等待人工确认。
- 发现非本切片产生的改动时，保留并绕开；无法绕开时停止。

### 3.3 技术规则

- 当前项目是静态页面与原生 JavaScript 架构。首版传导图优先使用原生 SVG，不为可视化引入大型前端框架。
- 新模块需同时支持浏览器和 Node 测试，沿用现有 UMD / CommonJS 兼容方式。
- 领域计算、数据适配、界面渲染必须分层，禁止继续把全部逻辑堆入 `micro_dashboard.js`。
- UI 可以整体重做，但采用“新工作台并行入口 → 验收 → 切换默认入口”的迁移方式；这不是保留旧布局，而是降低重构风险。
- 所有阈值、状态与公式必须可解释、可测试、可在配置中追溯，不得生成无法拆解的黑箱总分。

## 4. 目标信息架构

### 4.1 页面骨架

```text
┌──────────────────────────────────────────────────────────────┐
│ 情景 / 数据时间 / 来源健康 / P3 状态 / 当前研究状态          │
├──────────────┬──────────────────────────┬────────────────────┤
│ 左：研究导航 │ 中：微观传导图           │ 右：证据检查器     │
│ 情景         │ 机制—因子—状态—资产      │ 节点/边/路径证据   │
│ 资产         │ 支持/冲突/缺失/过期      │ 来源/时间/缺口     │
│ 路径筛选     │ 路径高亮                 │ 失效条件           │
│ 当前问题     │                          │ 校准状态           │
├──────────────┴──────────────────────────┴────────────────────┤
│ 下：观察时间线 / 24h / 48h / 72h 结果 / 路径状态变化         │
└──────────────────────────────────────────────────────────────┘
```

### 4.2 五种工作模式

1. `Overview`：当前情景、核心路径、总体研究判断与最大缺口。
2. `Transmission`：完整路径与边级验证状态。
3. `Evidence`：来源、时间戳、原生/代理、冲突与缺失原因。
4. `Observation`：P3 观察、事件账本与 24/48/72h 结果。
5. `Data Quality`：覆盖、新鲜度、多来源一致性、原生占比与降级来源。

### 4.3 研究语言

优先使用：

- 支持 / 反驳 / 中性 / 证据不足；
- 路径确认 / 部分确认 / 冲突 / 混合 / 过期 / 失效；
- 风险姿态、暴露压力、风险预算上下文；
- 人工决策准备度。

逐步隔离或改名：

- `reduce_long_risk` / `reduce_short_risk` → `risk_posture`；
- `target_delta_pct` → `exposure_pressure`；
- `risk_budget_pct` → `risk_budget_context`；
- 任何“可执行建议” → “研究判断 / 决策准备度”。

兼容字段可在适配层暂时保留，但 UI 与新契约不再把它们当作产品主语言。

## 7. 文件职责目标

```text
micro-layer/
  contracts/       # schema、枚举、示例夹具
  domain/          # 质量、状态、因子、市场状态、传导路径、校准判断
  application/     # 把领域结果组装为工作台 view model
  adapters/        # 当前快照、宏观上下文、事件账本、连续数据读取
  ui/              # SVG 图、检查器、导航、时间线与样式
  tests/           # 契约、领域、适配、浏览器状态测试
```

迁移原则：

- `micro_layer.js` 继续作为现有计算入口，先通过 adapter 接入；不要大爆炸重写。
- `micro_dashboard.js` 不再承接新的领域计算。
- `micro_macro_bind.js`、`micro_event_ledger.js`、`micro_calibration_gates.js` 优先复用，通过测试决定是否迁入新目录。
- 任何模块移动都要先补测试，再移动，再验证浏览器脚本顺序。
