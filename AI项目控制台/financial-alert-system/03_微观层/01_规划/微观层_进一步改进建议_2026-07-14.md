# 微观层进一步改进建议

## 一、总体判断

微观层已经完成“功能搭建”阶段：页面可以加载、离线验收通过、多交易所证据可以聚合、安全边界有效。下一阶段不应继续堆叠更多界面模块，而应把重心转向：

1. 连续数据是否可靠。
2. 历史事件证据是否真实、可追溯。
3. 校准规则是否经得住样本外和成本压力验证。
4. `abstain` 是否能向用户解释清楚。
5. 微观证据是否真正服务于宏观推导验证。

改进目标不是增加交易功能，而是把微观层从“可运行的分析页面”提升为“可持续积累证据的研究系统”。

## 二、当前进度基线

截至 2026-07-14：

| 项目 | 当前状态 | 说明 |
|---|---|---|
| 离线功能验收 | 8/8 通过 | 核心逻辑、页面、校准、组合、安全与契约均通过 |
| 浏览器验收 | 通过 | 7 个标的加载，主要页面区块完整 |
| 平均证据质量 | 约 89% | 属于当前页面样本，不代表长期稳定性 |
| 当前行动结论 | 7/7 `abstain` | 证据不足时拒绝给方向，属于正确安全行为 |
| 多交易所覆盖 | Binance、Bybit、Bitget 正常 | OKX 返回空值并已降级 |
| ETF 资金流 | 不稳定 | 当前存在外部来源 502 与可信度缺口 |
| 清算热图 | 证据不足 | 付费/高质量来源尚未接入 |
| 校准门禁 | 60 组规则中严格门禁 0 组通过 | 旧口径 6 组“有效”需要降级，54 组继续暂定 |
| P3 观察 | active_observation_only | 已有 3 次观察，28 天窗口刚开始 |
| P3 最早观察结束 | 2026-08-11 | 达到时间不等于自动通过 |
| 自动交易 | 禁用 | advisory-only、杀开关开启、订单提交禁止 |
| 发布状态 | RC 阻塞 | 连续数据、历史校准和观察窗口尚未完成 |

注意：当前工作区仍有未提交改动。最新验收产物可以用于研究判断，但在重新形成干净提交并完整回归前，不能视为正式发布基线。

## 三、改进原则

### 1. 证据优先于结论

- 先说明数据来自哪里、何时更新、是否冲突，再给出分析结果。
- 缺失数据必须保留 `missing_reason`，不得用默认值掩盖。
- 代理数据必须明确标注为 proxy，不能与原生数据混用。
- 来源可信不等于结论有效，来源质量与预测有效性分开评估。

### 2. 克制优先于强制输出

- `abstain` 是正式结论之一，不是错误占位符。
- 证据冲突、校准不足、数据过期时，应保持观察而不是强制生成多空方向。
- 页面应解释为什么 abstain，以及需要什么证据才能解除。

### 3. 研究与执行继续隔离

- 微观层继续只提供研究建议和证据核验。
- 不增加交易所密钥、订单接口、仓位执行器或自动授权。
- 执行门禁只作为安全边界测试，不作为未来自动交易预留。

## 四、P0：连续数据与运行可靠性

这是当前最高优先级，也是 P3 观察能够成立的前提。

### 改进项

1. 完成 [[7x24数据采集能力_执行计划_2026-07-14]] 的 S0–S2。
2. 修复 `quant-crypto-watcher` 包结构与 Compose 可重建配置。
3. 对 BTC、ETH、SOL 连续采集价格、资金费率、OI 与期现基差。
4. 每条数据记录来源时间、接收时间、采集时间和新鲜度。
5. 加入断线重连、指数退避、检查点续采和幂等写入。
6. 采集器故障时，不让研究层继续展示“正常”状态。
7. 验证 `FinancialAlert-MicroP3-8h` 第一次自动运行，并保存任务结果。

### 验收标准

- 连续运行 24 小时，无重启风暴。
- Docker 重启后可以自动恢复采集。
- P0 数据重复率为 0。
- 数据缺口、延迟和最后成功时间可见。
- P3 每 8 小时任务至少连续成功运行 3 次。
- 采集失败时页面显示 stale/missing，而不是沿用旧数据冒充实时数据。

## 五、P0：修复来源缺口

### OKX

当前 OKX 返回空值，系统虽然正确降级，但需要完成以下处理：

1. 区分接口成功但数据为空、限流、字段变更、合约代码错误和网络错误。
2. 保存 HTTP 状态、响应摘要和错误分类，禁止只记录 `empty`。
3. 建立 OKX 连续成功/失败计数与熔断恢复条件。
4. OKX 恢复前保持三交易所聚合，不静默伪造第四来源。

验收：连续 20 次采集记录中，成功率、失败原因和恢复行为均可解释。

### ETF 资金流

当前来源存在 502，不宜把单一网页抓取作为核心证据。

建议：

1. 将 ETF 资金流从“核心实时因子”降级为“辅助日频证据”。
2. 接入至少一个可追溯的备用来源。
3. 保存发布日期、适用交易日和是否最终修订。
4. 当日数据未发布时显示 `pending_release`，而不是 0。

### 清算热图

1. 明确区分交易所原生清算流、公开聚合数据和付费热图。
2. 未获得可靠热图时，页面显示“缺少空间分布证据”。
3. 不用单笔清算量或推测值冒充热图结构。
4. 付费数据源的采购应单独评估成本、授权和历史覆盖，不作为当前 P0 阻塞。

## 六、P1：建立原生事件证据账本

严格校准门禁 0/60 通过的核心原因之一，是大量历史样本仍依赖汇总或合成脚手架，缺少原生事件证据。

### 建议结构

每个候选事件保存：

- `event_id`
- `factor`
- `symbol`
- `regime`
- `event_started_at`
- `source_timestamp`
- `entry_mark_price`
- `24h/48h/72h exit_mark_price`
- `aligned_return`
- `source_legs`
- `source_trust`
- `raw_evidence_refs`
- `overlap_group`
- `cost_assumption_bps`
- `resolution_status`

### 改进项

1. 只用真实采集事件进入原生证据账本。
2. 现有合成样本保留为开发测试，不进入正式校准结论。
3. 对重叠事件建立分组，避免同一行情被重复计算。
4. 结果到期后自动解析 24/48/72 小时收益，但不自动改写正式校准簿。
5. 每次校准保留数据版本、代码提交和规则版本。

### 验收标准

- 任一事件可以追溯到原始采集记录。
- 入场与退出价格都有来源和时间戳。
- 合成样本与原生样本可以明确分离。
- 重跑相同数据得到一致结果。

## 七、P1：重构校准门禁

当前 60 组候选中，严格门禁 0 组通过。建议保留这一保守结果，不降低标准换取“有信号”。

### 正式门禁

候选规则必须同时满足：

1. 基础样本量与方向稳定性达标。
2. 使用原生事件证据。
3. 滚动样本外验证通过。
4. 去除事件重叠后仍有效。
5. 10bps 成本压力后仍具备正向净期望。
6. 跨时间窗口或市场状态不过度集中。
7. 来源质量与结论有效性分别通过。

### 状态定义

- `provisional`：可观察，不进入正式规则簿。
- `candidate`：原生证据达到最低要求，等待样本外验证。
- `validated`：全部严格门禁通过。
- `demoted`：旧规则在严格门禁下不再成立。
- `retired`：确认失效或口径已废弃。

### 当前建议

- 6 组旧“有效”规则先标记为 `demoted_pending_review`。
- 54 组未通过规则保持 `provisional`。
- 在原生事件样本不足前，不新增正式有效规则。
- 正式校准簿继续保持不变，直到 P3 观察和人工签核完成。

## 八、P1：完善 P3 观察

当前 P3 只有 3 次观察，仍处于起步阶段。

### 改进项

1. 每 8 小时观察任务必须自动运行，不依赖人工启动。
2. 每次观察记录数据覆盖、来源缺口、候选状态和非安全约束。
3. 到期结果按照 24/48/72 小时分批解析。
4. 每日生成运行摘要，每周生成证据变化报告。
5. 观察期间只更新候选证据，不修改正式校准簿。
6. 增加停止条件：连续采集故障、数据时间错位、正式簿哈希变化、订单提交尝试。

### 28 天结束条件

到 2026-08-11 只代表满足最短时间，不代表自动通过。结束时还需检查：

- 有效观察次数是否足够。
- 数据源覆盖是否稳定。
- 候选事件是否有足够到期结果。
- 是否存在长时间数据缺口。
- 样本是否过度集中于单一市场状态。
- 安全审计是否始终通过。

## 九、P2：优化研究界面

下一版界面建议从“指标陈列”转为“研究问题回答”。

### 首页顺序

1. 当前微观状态。
2. 对宏观推导的支持、反驳或证据不足。
3. 最大不确定性。
4. 数据新鲜度与来源覆盖。
5. 重点标的与关键因子。
6. 证据链和详细数据。

### `abstain` 解释卡

每次 abstain 应显示：

- 直接原因。
- 是数据缺失、因子冲突、校准不足还是组合上下文不足。
- 哪些证据已经具备。
- 哪些证据仍缺失。
- 下一次可能更新的时间或条件。

### 区分系统故障与研究克制

界面至少区分：

- `healthy_abstain`：系统正常，研究证据不足。
- `degraded_abstain`：部分来源降级，结论受限。
- `failed_no_decision`：系统故障，无法形成有效观察。

这样可以避免用户把“没有方向结论”误解为“功能坏了”。

## 十、P2：加强宏观—微观桥接

微观层的核心用途是验证宏观推导，而不是独立制造交易观点。

### 改进项

1. 每次微观观察绑定宏观场景 ID、路径 ID 和宏观上下文时间戳。
2. 明确输出 `support`、`contradict`、`neutral`、`insufficient`。
3. 宏观数据过期时，不把旧场景作为当前上下文。
4. 保存“宏观预期 → 微观可观察量 → 实际证据 → 偏差原因”的链条。
5. 对同一宏观场景建立时间序列，观察微观证据如何变化。

### 示例

> 宏观假设：流动性收紧 → 预期微观表现：杠杆拥挤下降、资金费率回落、风险资产 OI 收缩 → 实际证据：部分支持 / 冲突 / 不足 → 结论：继续观察或修正传播路径。

## 十一、P2：数据质量和来源可信度展示

当前平均质量分数不足以解释问题，建议拆成：

- 覆盖率。
- 新鲜度。
- 多来源一致性。
- 原生数据占比。
- 历史连续性。
- 校准适用性。
- 当前降级来源。

综合分数仍可保留，但必须能够展开查看构成，避免“89%”造成过度确定感。

## 十二、P3：研究治理与发布门禁

### Harness 检查

在现有研究质量流水线中继续加入：

- 数据契约测试。
- stale/missing/conflict/abstain 四状态回归。
- 事件幂等与到期结果解析测试。
- 校准簿不可变测试。
- 安全边界与禁止订单提交测试。

### 现场门禁

Harness 通过不能替代：

- 24 小时连续采集验收。
- 72 小时稳定性观察。
- 28 天 P3 观察。
- 人工研究复核。

## 十三、执行顺序

### 关键路径

> 7×24 S0 修复 → 连续数据 S1/S2 → P3 自动任务验证 → 原生事件账本 → 严格校准 → 宏观桥接 → 28 天复核

### 第一批立即执行

1. 修复 7×24 采集主链。
2. 手动触发并验收一次 P3 周期，随后验证连续 3 次自动运行。
3. 修复 OKX 错误分类与空数据诊断。
4. 将 ETF 资金流降级为辅助证据并增加备用来源设计。
5. 固化 60 组校准规则的严格门禁状态，防止旧“有效”标签继续误导页面。

### 第二批

1. 建立原生事件账本。
2. 自动解析 24/48/72 小时结果。
3. 增加每日摘要和每周证据报告。
4. 重构 abstain 与数据质量解释界面。

### 第三批

1. 完成宏观场景与微观观察的双向追溯。
2. 完成 28 天观察复核。
3. 对候选规则执行保留、降级或淘汰决策。
4. 重新执行研究级 RC 门禁。

| 阶段 | 当前状态 | 下一验收点 |
|---|---|---|
| 功能与页面 | 已通过 | 干净提交上的完整回归 |
| 连续数据 | 未成立 | 24 小时连续采集 |
| P3 自动观察 | 已配置、未充分验收 | 连续 3 次自动成功 |
| 原生事件证据 | 不足 | 事件可完整追溯 |
| 严格校准 | 0/60 通过 | 不降低标准，积累原生样本 |
| 研究界面解释力 | 可用、待增强 | abstain 三态与质量拆分 |
| 宏观—微观桥接 | 已接入基础上下文 | 场景/路径/观察全链追溯 |
| 研究级 RC | 阻塞 | 连续数据 + 28 天观察 + 人工签核 |

## 十五、不做事项

- 不增加自动交易功能。
- 不连接订单提交接口。
- 不因 7×24 采集而扩大为交易执行系统。
- 不为了产生“可行动信号”而降低校准标准。
- 不把合成样本包装成真实历史证据。
- 不把单次浏览器通过写成长期稳定性通过。
