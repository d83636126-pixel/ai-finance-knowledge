# 微观层仪表盘：研究总览恢复与重构执行计划

## 0. 计划定位

本计划接续 [[AI项目控制台/financial-alert-system/03_微观层/01_规划/微观层整体重构_Cursor逐步执行清单_2026-07-14|C0—C9 微观层整体重构清单]]，用于修正 C8/C9 后默认工作台**功能存在但首屏表达过度简化**的问题。

当前 C0—C9 已完成代码与页面功能验收，不能重新写成“尚未实现”；但研究级 RC 仍受24h、72h与P3 28天时间门禁约束，不能因本次界面调整而提前宣告通过。

本轮目标不是恢复旧式指标堆叠，也不是向自动交易延伸，而是形成：

> 一屏研究总览 → 发现异常与证据缺口 → 进入传导、证据、观察和质量深页 → 返回人工研究结论

## 1. 当前判断与现场基线

### 1.1 已确认的现状

- 正式源码：`F:\financial-alert-system`。
- 项目总仪表盘 `index.html` 没有明显缩减，仍保留事件时间线、评分、AI信号、系统状态、绩效、情绪与相关性模块。
- 被明显压缩的是 `micro-layer/index.html` 默认页面：旧版多资产总览、KPI、宏观映射、P3状态、因子、趋势和来源可信度被删除、折叠或分散到五个模式。
- 当前工作台保留五个模式：Overview、Transmission、Evidence、Observation、Data Quality。
- 当前测试结果：
  - `npm run smoke:micro-workbench`：10/10通过；
  - `npm run smoke:micro-page`：默认工作台与旧版回退入口通过。
- 旧版卡片页仍位于 `micro-layer/index_legacy.html`，可作为功能映射和回退基线。
- 当前页面能够运行，但首屏信息密度偏低，且单独出现 `confirmed / support` 时，没有同步突出数据来源、采集状态、时效、反证与证据边界。

### 1.2 核心问题

1. 用户进入默认页后，只能看到单一路径摘要，无法快速判断整个微观层当前状态。
2. BTC、ETH、SOL不能在同一屏横向比较，只能下拉切换。
3. 数据来源健康、P3状态、连续采集状态和更新时间没有进入首屏决策层。
4. 旧版已有的数据质量、因子和趋势信息被隐藏得过深。
5. Observation 在没有充分账本结果时过于稀疏，大量留白造成“功能被削弱”的感受。
6. 英文模式名称和偏技术字段增加理解成本。
7. “路径确认”容易被误读为“可交易结论”，需要与证据质量和人工复核边界绑定显示。

## 2. 不可突破的边界

- 项目继续以**宏观推导 + 微观分析**为中心。
- 不增加交易所私钥、下单、订单路由、自动仓位、目标仓位或授权按钮。
- 保持 `advisory-only`、`executable=false`、`human_review_required=true`。
- 数据过期、缺失、冲突或校准不足时，必须允许 `abstain / insufficient`。
- 不降低校准门槛，不改写正式校准簿。
- 不把 fixture、离线样本或代理证据伪装成实时原生证据。
- 不因界面验收通过而宣告24h、72h、P3 28天或研究级 RC 通过。
- 新首页验收完成前，不删除 `index_legacy.html`。

## 3. 目标信息架构

```text
┌──────────────────────────────────────────────────────────────┐
│ 情景 / 数据时间 / 实时性 / 来源健康 / P3状态 / 研究状态      │
├──────────────────────────────────────────────────────────────┤
│ 当前结论：支持 / 反驳 / 混合 / 证据不足                      │
│ 依据 · 反证 · 最大缺口 · 有效期 · 下一项核验                 │
├──────────────────────────────────────────────────────────────┤
│ BTC                 │ ETH                 │ SOL               │
│ funding / OI / basis│ funding / OI / basis│ funding / OI / basis│
│ 质量 / 来源 / 状态  │ 质量 / 来源 / 状态  │ 质量 / 来源 / 状态│
├──────────────────────────────────────────────────────────────┤
│ 核心传导路径摘要    │ 数据与采集健康       │ 最近状态变化       │
├──────────────────────────────────────────────────────────────┤
│ 研究总览 | 传导路径 | 证据检查 | 观察回放 | 数据质量          │
└──────────────────────────────────────────────────────────────┘
```

首页只负责回答五个问题：

1. 当前研究情景是什么？
2. 微观证据支持、反驳还是不足？
3. BTC、ETH、SOL分别处于什么状态？
4. 数据是否新鲜、来源是否健康、P3与采集是否连续？
5. 当前最大证据缺口和下一步核验是什么？

## 4. Cursor 逐步执行切片

### C10：冻结现状与建立功能映射

任务：

- 保存当前默认工作台与旧版卡片页的浏览器截图。
- 建立“旧版能力 → 当前模式 → 新首页位置 → 验收状态”矩阵。
- 记录当前五模式、数据接口、DOM入口和回退入口。
- 明确哪些旧能力迁移、合并、仅留深页或正式废弃。

不得：

- 不修改领域判断、阈值、校准或采集逻辑。
- 不删除旧版页面。

验收：

- 功能映射没有“找不到去向”的核心研究能力。
- 明确改动文件范围和非本轮文件范围。

### C11：重建首页研究总览

任务：

- 将默认 `Overview` 改为中文“研究总览”。
- 首屏恢复以下信息：
  - 宏观情景、核心路径和研究命题；
  - 研究立场与人工准备度；
  - 数据时间、有效期和实时/离线状态；
  - 最大证据缺口与解除条件；
  - P3观察状态与连续采集状态；
  - 最近一次状态变化。
- 减少无信息留白；未知字段显示“尚无证据”，不得静默消失。

建议文件：

- `micro-layer/index.html`
- `micro-layer/ui/workbench.css`
- `micro-layer/ui/workbench_app.js`
- 可新增 `micro-layer/ui/overview_dashboard.js`

验收：

- 在1366×768视口内，核心情景、结论、时效、来源健康和最大缺口无需进入二级模式即可读取。

### C12：恢复多资产横向比较

任务：

- 在首屏同时展示BTC、ETH、SOL，不再只依赖下拉切换。
- 每个资产至少展示：
  - 资金费率；
  - OI变化；
  - 期现基差或明确的缺失标记；
  - 价格变化；
  - 数据质量；
  - 来源可信度；
  - 市场状态；
  - 证据立场。
- 点击资产卡片后，切换深页焦点，不在卡片内重新计算路径结论。

验收：

- 三个资产在同一数据时间下可比较。
- 数据源异常与市场异常能够被明显区分。
- 缺失或代理数据不能显示为健康原生数据。

### C13：修正结论表达与证据边界

任务：

- 禁止孤立显示 `confirmed` 或 `support`。
- 所有结论卡必须成组显示：
  - 路径状态；
  - 研究立场；
  - 数据质量；
  - 数据时间与有效期；
  - 支持证据；
  - 反证；
  - 缺失证据；
  - 人工复核要求。
- 建立明确的状态文案：已确认、部分确认、混合、已反驳、证据不足、已过期、非实时样本。
- 对 stale、missing、conflict 和 fixture 使用醒目的状态色与说明。

验收：

- 任一结论均可追溯“为什么成立、什么会使它失效、证据何时过期”。
- 页面不产生“路径确认等于交易信号”的歧义。

### C14：强化五个深度模式

任务：

- 全部改为中文：
  - Overview → 研究总览；
  - Transmission → 传导路径；
  - Evidence → 证据检查；
  - Observation → 观察回放；
  - Data Quality → 数据质量。
- 传导路径：补充方向、预期时滞、失效条件和逐边证据。
- 证据检查：展示来源、时间、原始引用、角色、缺失原因和代理标记。
- 观察回放：展示状态变化、24/48/72h槽位和“当时可知 / 事后结果”隔离。
- 数据质量：展示来源腿健康、覆盖率、新鲜度、一致性、原生证据比例、降级来源和付费数据缺口。
- 五个模式继续复用同一 view model，不允许各自重算结论。

验收：

- 从首页任意异常可在两次点击内进入对应证据。
- 深页结论与首页完全一致。

### C15：视觉层级、响应式与回退

任务：

- 优化大面积空白、字号、中文字段、状态色和信息密度。
- 保留宽屏三栏；普通笔记本转为两栏；移动端转为单栏列表。
- 首屏突出异常、过期、冲突和缺失；正常项降低视觉噪声。
- 保留“旧版卡片页”回退入口，但降低视觉优先级。
- 新版出现契约或加载失败时，显示可理解错误，不使用静默fixture掩盖故障。

验收：

- 1366×768、1440×900和移动宽度均无严重遮挡、横向溢出或不可点击区域。
- 浏览器刷新、资产切换和五模式切换均稳定。

### C16：逐项验收、切换与归档门禁

必须形成逐项验收矩阵：

| 功能 | 验收方式 | 通过条件 |
|---|---|---|
| 首页研究总览 | 浏览器检查 | 一屏回答五个核心问题 |
| 多资产比较 | BTC/ETH/SOL实测 | 同屏、同时间、缺失诚实 |
| 宏观—微观桥接 | 场景检查 | 预期→可观察→证据→立场完整 |
| 传导图 | 节点/边点击 | 方向、时滞、失效条件可见 |
| 证据检查 | 原生/代理/合成样本 | 来源与角色不混淆 |
| 数据质量 | stale/missing/conflict | 均能触发正确降级 |
| P3与采集 | 当前产物检查 | 状态、时间窗和失败信息可见 |
| 观察回放 | 24/48/72h槽位 | 当时结论与事后结果隔离 |
| 安全边界 | DOM与测试 | 无下单/授权/目标仓位入口 |
| 旧版回退 | 导航测试 | 新版验收前始终可访问 |

最低验收命令：

```powershell
npm run smoke:micro-workbench
npm run smoke:micro-page
npm run smoke:micro-transmission-engine
npm run smoke:micro-collect-snapshot
npm run accept:micro-redesign
npm run accept:micro-s3:offline
npm run check
```

切换条件：

- 功能矩阵全部通过；
- 浏览器实测无阻断；
- 没有订单或授权控件；
- fixture、离线、过期和缺失状态均能正确显示；
- 旧版核心研究能力均已迁移或有明确废弃记录。

归档条件：

- 新首页稳定运行一个观察周期；
- 用户确认新版信息密度与研究流程可接受；
- 才能将 `index_legacy.html` 移入历史归档。

- 旧版观察期、对照验收与归档。

## 6. Cursor 执行纪律

每次只执行一个切片。开始前必须报告：任务编号、目标、预计修改文件、不会修改的内容和验收命令。结束后必须报告：实际修改文件、测试结果、浏览器证据、未解决问题、`git status --short`和是否建议进入下一切片。

禁止：

- `git reset --hard`、`git checkout --`、`git clean`；
- 自动push；
- 使用 `git add .`；
- 覆盖非本切片的dirty改动；
- 用测试通过替代24h、72h与28天真实时间门禁；
- 为了界面完整而伪造实时数据或确定性结论。

## 7. 完成定义

新仪表盘必须让研究者在一分钟内回答：

1. 当前情景和核心传导路径是什么？
2. 微观证据支持、反驳还是不足？
3. 三个主要资产的状态有何差异？
4. 数据是否可信、是否新鲜、采集是否连续？
5. 最大证据缺口是什么，下一步核验什么？

满足以上条件，只能声明“仪表盘研究功能验收通过”。24h、72h、P3 28天和研究级 RC 仍按原时间门禁独立签署。
