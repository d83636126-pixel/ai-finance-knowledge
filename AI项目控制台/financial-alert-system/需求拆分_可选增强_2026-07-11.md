# financial-alert-system：可选增强需求拆分

## 背景

P1 与 P2 主线已经闭环。当前进入可选增强阶段，本轮只拆分以下两项：

1. 历史数据事件加厚与历史盲测（优先）。
2. 事前 / 事后结果常驻面板最小版（其次）。

本轮不包含 P3 推送、Cron、7×24 扫描和 Scanners 后端化。

> **2026-07-11 实施结果**：第一阶段两项均已落地（见 [[任务进度]]）。第二阶段管理能力不阻塞验收。

## 需求一：事前 / 事后结果常驻面板

### 目标

用户生成事前报告、事后报告或事后对比后，无需停留在当前弹窗或事件卡中，也能随时回看最近结果，并能从结果返回原事件或关联的图谱推演路径。

### 用户场景

- 用户从日历生成事前报告，切换到图谱后仍可回看报告。
- 用户从 Inbox 生成事后对比，可同时查看该事件已有的事前结果。
- 用户从 Dashboard 将 Alert 加入 Inbox，完成分析后可在统一位置查看结果。
- 用户刷新页面后，最近生成的分析结果仍然存在。

### 第一阶段最小范围

- [x] 在 Propagation 页面增加“分析结果”常驻区域。
- [x] 统一接收日历、Inbox、Dashboard 入库三类来源的分析结果。
- [x] 生成事前报告、事后报告或事后对比后，自动写入常驻区域。
- [x] 结果卡展示事件名称、事件来源、报告类型、生成时间和摘要。
- [x] 展示结果关联的图谱事件、节点和推演路径。
- [x] 支持在同一事件的事前、事后、对比结果之间切换。
- [x] 支持从结果卡返回原事件。
- [x] 支持从结果卡进入对应的图谱推演。
- [x] 新结果置顶，明确标识“事前”“事后”“对比”。
- [x] 页面刷新后保留最近结果。
- [x] 无 DeepSeek Key、接口失败或结果为空时显示明确状态与重试入口。

### 第二阶段可选范围

- [x] 支持清除单条结果（面板「删除」）。
- [ ] 支持清空全部结果与更完整的容量/失败恢复体验。

### 数据要求

日历、Inbox 和 Dashboard 生成的结果应使用同一数据结构，至少包含：

| 字段 | 说明 |
|---|---|
| `result_id` | 结果唯一标识 |
| `event_id` | 关联事件标识 |
| `event_name` | 事件名称 |
| `source_type` | `calendar`、`inbox` 或 `alert` |
| `report_type` | `pre`、`post` 或 `compare` |
| `created_at` | 生成时间 |
| `summary` | 结果摘要 |
| `content` | 完整报告内容 |
| `graph_event` | 关联的图谱事件 |
| `node_ids` | 关联节点列表 |
| `path_context` | 图谱推演上下文 |
| `status` | `success`、`failed` 或 `empty` |

第一版使用浏览器本地存储，不增加新的后端依赖。应设置合理的数量或容量上限，避免长期累积影响页面性能。

### 交互约束

- 常驻区域不能长期遮挡图谱主视图。
- 推荐使用可收起的右侧栏或底部抽屉。
- 关闭面板不等于删除结果。
- 切换事件或页面 Tab 时，当前结果不应丢失。
- 内容较长时，应提供摘要视图和完整内容展开入口。
- 保持现有日历、Inbox、Case 和图谱操作方式不变。

### 异常处理

- 未配置 DeepSeek Key：展示配置提示，不创建伪结果。
- 请求失败：保留事件上下文并提供重试按钮。
- 返回空内容：标记为空结果，不覆盖该事件已有的成功结果。
- 本地存储不可用：当前会话仍可查看结果，并提示刷新后可能丢失。
- 关联节点不存在：报告仍可查看，但图谱跳转入口应禁用并说明原因。

### 验收标准

- [x] 从日历生成事前报告后，可在常驻面板中回看。
- [x] 从 Inbox 生成事后对比后，可切换查看同一事件的事前结果。
- [x] Dashboard 入库后的分析结果进入同一个常驻面板。
- [x] 点击“查看推演”可正确打开对应图谱路径。
- [x] 点击“返回事件”可定位到日历或 Inbox 中的原事件。
- [x] 硬刷新页面后，最近结果的内容、时间、来源保持正确。
- [ ] （第二阶段可选）单条删除和全部清空功能正常。
- [x] 无 Key、接口失败、空结果场景均有明确反馈。
- [x] 不破坏现有日历、Inbox、Case 和图谱功能。

### 暂不包含

- 多设备同步分析结果。
- 服务端长期存档。
- 多用户共享或权限管理。
- 自动生成 Obsidian 报告笔记。

## 需求二：历史数据事件加厚与历史盲测

### 目标

扩充 `MAJOR_EVENTS` 中的历史宏观事件，为 NFP、CPI 等 Case 提供更密集、可追溯的历史样本，用于路径复盘、事前 / 事后对比以及后续回测。历史回测必须区分复盘与盲测，禁止把实际公布值或事后行情泄漏给“事前”报告。

### 模式定义（强制）

#### 复盘模式

- 允许读取 `actual`、修正值和事后市场表现。
- 用于解释事件发生后的实际传导与验证结果。
- 页面必须明确显示“复盘”，不得称为事前预测。

#### 历史盲测模式

- 以事件公布时间作为信息截止点（as-of cutoff）。
- 只允许读取截止时间之前已公开的 `forecast`、`previous`、当时可见的宏观数据与图谱版本。
- 必须隐藏 `actual`、后续修正值、事后行情以及由这些数据推导的 evidence 状态。
- 先保存盲测报告，再解锁实际值并生成事后对比。
- 报告中记录截止时间、可用字段和图谱版本，保证结果可复现。

### 第一批范围

- [x] 优先补充 NFP 历史事件。
- [x] 优先补充 CPI 历史事件。
- [x] 每类先形成一批连续、可验收的小样本。
- [x] 具体样本数量在检查现有数据结构和页面容量后确定。
- [ ] 第一批完成后再评估是否扩充 PPI、FOMC 等事件。

### 单条事件字段

每条历史事件至少包含：

| 字段 | 说明 |
|---|---|
| `event_id` | 唯一且稳定的事件标识 |
| `event_type` | `NFP` 或 `CPI` |
| `event_name` | 用户可读的事件名称 |
| `released_at` | 实际公布时间，包含时区 |
| `period` | 数据对应统计周期 |
| `forecast` | 市场预期值 |
| `actual` | 实际公布值 |
| `previous` | 前值 |
| `revised_previous` | 修正后前值；如无则明确为空 |
| `unit` | 数值单位 |
| `surprise` | 实际值与预期值的差异 |
| `direction` | 偏热、偏冷或符合预期 |
| `source` | 数据来源名称 |
| `source_url` | 可追溯的来源地址 |
| `graph_event` | 对应的图谱事件类型 |
| `primary_nodes` | 主要影响节点 |
| `observation_window` | 事后观察窗口 |
| `data_status` | `confirmed`、`partial` 或 `missing` |
| `as_of_cutoff` | 历史盲测的信息截止时间 |
| `graph_version` | 生成报告时使用的图谱版本 |
| `test_mode` | `replay` 或 `blind` |

### 数据规则

- [x] 复用现有 `MAJOR_EVENTS` 数据结构，不建立平行事件体系。
- [x] 统一日期格式、时区和数值单位。
- [x] 事件 ID 不得重复，且不能因展示名称调整而改变。
- [x] 缺失数据使用明确的空值和状态，不使用 `0` 代替。
- [x] 修正值与初值分开保存，不覆盖原始公布信息。
- [x] 口径变化必须添加说明，避免跨期直接误比较。
- [x] surprise 的计算规则在同一事件类型内保持一致。
- [x] 优先采用官方或已确认的数据来源，并保留来源信息。
- [x] 历史盲测读取数据时强制应用 `as_of_cutoff`，不得访问截止时间后的字段。
- [x] 盲测报告必须先落盘或持久化，再显示 actual 和事后对比。

### 页面与流程接入

- [x] 日历能按日期显示新增历史事件。
- [x] 日历能按 NFP、CPI 类型筛选新增事件。
- [x] 新增事件能够进入现有事前报告流程。
- [x] 新增事件能够进入现有事后报告和对比流程。
- [x] Case 库能够选择新增 NFP、CPI 样本。
- [x] Case 切换历史样本后能够生成已有的主路径。
- [x] 结果常驻面板能够接收历史事件的分析结果。

### 数据校验

- [x] 校验必填字段是否完整。
- [x] 校验事件 ID 是否唯一。
- [x] 校验时间格式和时区是否合法。
- [x] 校验 forecast、actual、previous 的单位是否一致。
- [x] 校验 surprise 与 direction 是否符合计算规则。
- [x] 校验 graph event 和关联节点是否存在。
- [x] 对缺失值、修正值和异常值输出明确报告。
- [x] 自动检查盲测输入中是否混入 actual、修正后数据或事后行情。

### 验收标准

- [x] NFP、CPI 各完成一批连续历史样本。
- [x] 所有新增事件通过格式、必填字段和唯一性校验。
- [x] 新增事件可以在日历中按日期和类型找到。
- [x] 新增事件可以进入事前、事后与对比流程。
- [x] Case 库可以切换新增样本并生成主路径。
- [x] 缺失值、修正值、时区和单位可以明确辨识。
- [x] 每条数据均可追溯来源。
- [x] 历史盲测报告记录信息截止时间和图谱版本，且不存在数据穿越。
- [x] 现有 FOMC、NFP、CPI、PPI Case 回归通过。

### 暂不包含

- 一次性补齐全部历史年份。
- 未经校验的批量第三方数据直接入库。
- 构建新的数据抓取后端。
- 自动交易或收益率预测。

## 实施顺序

1. 检查 `MAJOR_EVENTS` 当前结构与现有历史样本。
2. 定义历史盲测的 `as_of_cutoff`、字段隔离和防穿越校验。
3. 确定 NFP、CPI 第一批样本数量和数据来源。
4. 分批录入数据，先完成盲测 → 解锁 actual → 事后对比的最小闭环。
5. 盘点现有事前、事后、对比结果的数据结构。
6. 统一三类入口的结果对象和本地持久化方式。
7. 完成常驻面板最小闭环：写入、回看、跳转、刷新保留。
8. 执行日历、Case、历史盲测、事后对比和常驻面板的联合回归。
9. 完成后更新项目的 `任务进度.md` 与 `会话交接.md`。

## 完成定义

以下核心条件满足后，本轮第一阶段可选增强才算完成；标记为第二阶段可选的管理功能不阻塞第一阶段验收：

- 事前、事后和对比结果能够统一写入并长期回看。
- 结果可返回原事件并进入对应图谱推演。
- 页面刷新不会丢失最近结果。
- NFP、CPI 历史样本完成第一批扩充并通过数据校验。
- 历史盲测不存在 actual、修正值或事后行情的数据穿越。
- 新增历史事件已接入日历、Case 和事前 / 事后流程。
- 现有主线功能回归通过，无新增阻塞。
