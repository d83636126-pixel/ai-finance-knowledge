# financial-alert-system：决策记录

### 2026-07-05 确认 target 分层规则

- 问题：多个 target 节点层级不清，预警输出缺少主次
- 选项：A) 所有 target 平等展示 / B) 分核心与扩展两层
- 选定：B
- 理由：第一版预警系统应优先输出核心资产篮子，扩展资产作为次级面板
- 影响范围：`propagation_graph.json`（target_tier / target_priority 字段）、前端 UI、报告排序
- 是否已被推翻：否

### 2026-07-05 USD / Treasury 纳入目标资产表达

- 问题：核心资产篮子中 USD / Treasury 没有对应的 target 节点
- 选项：A) 直接复用 usd_index 和 treasury_yield / B) usd_index 升级为 target，treasury_yield 保持 intermediate，新增 treasury_bond_return
- 选定：B
- 理由：收益率与债券价格方向相反，treasury_yield 不适合直接做 target
- 影响范围：`propagation_graph.json`（usd_index 升级、treasury_bond_return 新增）、图谱节点 113、边 252
- 是否已被推翻：否

### 2026-07-05 Nasdaq 升级为目标节点

- 问题：Nasdaq 是第一版核心输出资产但节点类型仍是 intermediate
- 选项：A) 保持 intermediate / B) 升级为 target
- 选定：B
- 理由：系统需要直接输出 Nasdaq 的风险方向，而不是只把 Nasdaq 当成 BTC/SPX 的中间传导变量
- 影响范围：`propagation_graph.json`（nasdaq.type = target）、target 数 5→6
- 是否已被推翻：否

### 2026-07-05 第一批边采用主干连接法

- 问题：新增 15 个节点后如何连边
- 选项：A) 全连接 / B) 只连主干路径（全球数据/事件 → 全球状态 → 核心资产篮子）
- 选定：B
- 理由：全连接会造成图谱噪音过大、解释困难。先建立主干，再逐步扩展细分边
- 影响范围：新增 32 条边、孤立节点从 15 降为 0、健康评分 77→88
- 是否已被推翻：否

### 2026-07-05 第一批节点先补节点不连边

- 问题：扩展图谱时应先补节点还是边
- 选项：A) 节点和边同时补充 / B) 先补节点，再单独设计边
- 选定：B
- 理由：节点补齐和边设计应分开进行，避免在传导逻辑尚未确认前乱连边
- 影响范围：图谱节点数 97→112、新增节点当前为孤立节点（预期状态）、健康评分 77（watch）
- 是否已被推翻：否

### 2026-07-05 确认图谱中心目标

- 问题：图谱应以什么为中心
- 选项：A) 以 BTC 为中心 / B) 以风险偏好为中心 / C) 以资产篮子为中心
- 选定：中心目标 = 全球风险状态，输出目标 = 资产篮子，重点资产 = BTC + Nasdaq + Gold + USD + Treasury
- 理由：以单一资产为中心会使全球宏观节点变成附属解释；以全球风险状态为中心可同时解释美元压力、利率压力、流动性、风险偏好、地缘冲击和多资产反应
- 影响范围：图谱设计方向、节点补齐优先级、边设计原则、预警输出结构
- 是否已被推翻：否（用户已确认）

### 2026-07-05 先拆分，不直接开发

- 问题：项目登记后应先做什么
- 选项：A) 直接修改源码 / B) 先明确大需求、小需求、当前阶段和验收标准
- 选定：B
- 理由：方便控制项目方向，避免 AI 跑偏
- 影响范围：所有后续开发动作需先有 Obsidian 文档再执行
- 是否已被推翻：否

### 2026-07-05 纳入 Obsidian 项目控制台

- 问题：financial-alert-system 如何纳入 AI 协作体系
- 选项：A) 继续保持独立 / B) 纳入 Obsidian 项目控制台
- 选定：B
- 理由：用户希望 Obsidian 成为项目流程控制层，让 AI 后续可以根据项目文档继续推进
- 影响范围：代码保留在原目录，Obsidian 只记录项目控制信息、需求、进度、决策和交接
- 是否已被推翻：否
