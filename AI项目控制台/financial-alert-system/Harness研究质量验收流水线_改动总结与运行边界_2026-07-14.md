# Harness 研究质量验收流水线：改动总结与运行边界

## 一、结论

Harness 已被接入为项目的**研究质量验收入口**，用于在代码变更后统一执行核心契约检查、微观层离线验收、研究工作台浏览器检查，并生成机器可读的质量门禁报告。

当前准确状态为：

- **流水线配置：已完成并提交**。
- **代码基线：本地已提交基线、GitHub `master`、Harness 仓库镜像均对齐到 `da8fdfb`**。
- **Harness 内部执行：尚未完成**，当前实例没有可用执行器/Runner，因此没有可作为证据的 Harness 运行记录。
- **研究级 RC 门禁：仍为阻塞状态**。离线功能检查通过也不能替代真实数据源稳定性、观察窗口和人工研究复核。
- **系统边界：仅提供研究与决策支持，不执行自动交易、不提交订单、不自动部署**。

因此，这次改动完成的是“质量检查制度化与可重复执行”，不是把项目改造成自动交易系统，也不是宣布研究级版本已经发布。

## 二、已完成的代码改动

### 1. 新增 Harness 流水线配置

配置文件：`F:\financial-alert-system\.harness\research-quality.yaml`

流水线名称：`research-quality-gate`

提交记录：

- `9b8ce6f ci: add research quality gate pipeline`
- `da8fdfb fix: close research workbench P1 P2 gates`

流水线包含以下阶段：

| 顺序 | 阶段 | 作用 | 主要命令 |
|---|---|---|---|
| 1 | 锁定依赖安装 | 按锁文件恢复一致的 Node.js 依赖 | `npm ci --ignore-scripts --no-audit --no-fund` |
| 2 | 核心契约检查 | 校准基础数据并检查项目核心契约 | `node scripts/phase4_blind_calibrate.js`、`npm run check` |
| 3 | 微观层离线验收 | 验证微观层 S3 在离线模式下的确定性行为 | `npm run accept:micro-s3:offline` |
| 4 | 工作台烟雾与功能验收 | 启动本地服务，执行浏览器烟雾检查和验收套件 | `npm run smoke`、`npm run acceptance` |
| 5 | 研究门禁报告 | 汇总检查结果，输出机器可读结论 | `node scripts/generate_research_quality_gate.js` |

### 2. 新增研究质量门禁报告生成器

脚本：`F:\financial-alert-system\scripts\generate_research_quality_gate.js`

该脚本读取以下验收产物：

- `artifacts/last_check.json`
- `artifacts/last_smoke.json`
- `artifacts/last_acceptance.json`
- `artifacts/micro_s3_acceptance.json`

并输出：

- `artifacts/research_quality_gate.json`

报告将结论拆成两层：

1. **功能门禁（functional gate）**：检查产物是否存在且均为通过。
2. **研究级 RC 门禁（research RC gate）**：即使功能门禁通过，也默认保持阻塞，直到现场数据与研究证据满足要求。

可能出现的核心决策值：

- `functional_pass_rc_blocked`：功能检查通过，但研究级发布仍被阻塞。
- `functional_failed`：至少一项功能检查失败。

## 三、门禁为何保持保守

当前报告固定保留以下研究级阻塞项：

- `live_source_stability_not_verified_by_offline_pipeline`：离线流水线不能证明真实数据源长期稳定。
- `p3_observation_window_not_verified`：尚未完成足够的观察窗口验证。
- `manual_research_review_required`：重要研究结论仍需人工复核。

这意味着：

> “自动检查全部通过”只能证明当前代码在既定测试条件下可重复工作，不能证明研究结论在真实市场环境中持续有效。

浏览器测试出现全 `abstain`（不作判断）时，也不应直接视为失败。如果输入证据不足，保持克制可能正是咨询型系统的正确行为。

## 四、与项目方向的关系

Harness 对本项目的具体用途是：

- 防止宏观推导、微观证据链和研究工作台在代码更新后发生静默退化。
- 将分散的检查统一为一条可追踪、可复现的质量流水线。
- 为每个提交形成明确的“功能通过 / 功能失败 / 研究 RC 阻塞”结论。
- 让 Obsidian 项目控制台能够引用结构化验收结果，而不是只依赖人工描述。
- 为以后接入真实数据源稳定性验证、观察窗口和人工签核预留门禁位置。

Harness **不负责**：

- 自动下单或交易执行。
- 生成交易授权。
- 将研究结论直接转换为仓位。
- 在未经人工确认时自动部署研究结论。
- 代替研究人员判断数据可信度与因果关系。

项目主线仍是：**宏观推导 → 微观证据核验 → 传播链分析 → 研究结论与不确定性表达**。

## 五、当前同步关系

当前采用受控、单向的职责划分：

1. 本地代码提交后推送到私有 GitHub 仓库。
2. Harness 使用与 GitHub 对齐的提交作为验收基线。
3. 验收摘要再写回本地 Obsidian 的项目控制台。

当前状态：

| 对象 | 状态 | 说明 |
|---|---|---|
| 本地已提交基线 | `da8fdfb` | 与 GitHub、Harness 对齐 |
| GitHub `master` | `da8fdfb` | 已推送 |
| Harness 仓库镜像 | `da8fdfb` | 已对齐 |
| 本地未提交工作区 | 存在后续改动 | 不属于上述已对齐基线，需另行验收与提交 |
| Obsidian 项目控制台 | 已写回状态 | 本地库已有改动，尚未自动公开发布 |

注意：Obsidian 不是代码反向同步源。修改研究笔记不会自动改写 GitHub 代码；更新 GitHub 也不会在没有同步动作时自动更新 Obsidian。

## 六、当前限制与风险

### 1. Harness 尚无真实运行证据

流水线已经在 Harness 中注册并启用，但当前本地 Harness 实例没有可用 Runner/执行器，因而不能在 Harness 内部真正启动容器步骤。当前不能把“配置已存在”写成“Harness 验收已通过”。

### 2. 严格浏览器验收仍需以最新运行结果为准

此前严格检查曾出现 11/13，通过项之外仍有界面可见性门禁未满足。后续本地工作区包含可能处理该问题的未提交改动，但在重新运行完整套件、形成证据并提交前，不能提前宣布关闭。

### 3. 凭据与自动化

若未来设置后台定时同步，需要持久化 Harness 凭据。默认建议优先采用“一键受控同步 + 临时令牌”，避免为了便利长期保存高权限令牌。

## 七、下一步验收顺序

1. 为 Harness 配置受限、隔离的 Runner，不向主机 Docker Socket 提供不必要的高权限访问。
2. 在提交 `da8fdfb` 上触发第一次 Harness 流水线运行。
3. 保存每一步日志和 `research_quality_gate.json` 作为验收证据。
4. 对本地未提交的界面修复重新执行核心检查、微观层离线验收、工作台烟雾检查与完整验收。
5. 只有最新严格套件全部满足时，才更新“功能门禁通过”的状态。
6. 真实数据源稳定性、P3 观察窗口与人工研究复核完成前，继续保持“研究级 RC 阻塞”。
7. 由人工确认后，再把新的验收摘要写回 Obsidian；是否推送 Obsidian 公共仓库另行决定。

## 八、验收口径

后续统一使用以下状态，避免歧义：

- **已配置**：文件和 Harness 对象已经存在。
- **已触发**：Harness 已产生一次运行记录。
- **功能验收通过**：最新完整检查全部通过，并有对应产物和日志。
- **研究级 RC 通过**：功能通过之外，真实数据源、观察窗口和人工复核均已完成。
- **已发布**：通过明确授权完成目标环境发布；本项目当前不默认执行此动作。
