# 7×24 数据采集能力执行计划

## 一、目标与边界

### 目标

为 `financial-alert-system` 建立可持续运行的数据采集层，连续获取微观市场、宏观指标和新闻事件数据，并形成可追溯、可补采、可验证的研究证据。

最终形成以下闭环：

> 数据源 → 持续采集 → 原始证据保存 → 标准化 → 数据质量检查 → 研究快照 → 宏观推导 / 微观分析 → Obsidian 状态记录

### 明确边界

- 系统只提供研究、观察、证据沉淀和决策支持。
- 不自动下单，不连接订单接口，不生成交易授权。
- 不将研究结论自动转换为仓位。
- 数据不足、过期或冲突时必须明确标记，不用估算值伪装成真实数据。
- Harness 负责代码与数据契约验收，不代替真实环境的连续运行观察。

## 二、当前事实基线

2026-07-14 现场检查结果：

| 对象 | 当前状态 | 判断 |
|---|---|---|
| `financial-alert-system` | 已有市场快照、宏观证据循环、微观层、P3 观察脚本 | 具备研究逻辑，但不是持续采集服务 |
| `FinancialAlert-MicroP3-8h` | 已注册，每 8 小时运行 | 适合阶段性观察，不等于 7×24 采集 |
| Redis | 正常运行 | 可作为队列、短期缓存和去重辅助 |
| TimescaleDB/PostgreSQL | 正常运行 | 可作为标准化时序数据主存储 |
| InfluxDB | 正常运行 | 可保留作运行指标和高频监控，不作为唯一事实源 |
| Worker / Beat | 正常运行 | 可复用为调度和后台处理基础 |
| RSSHub | 健康 | 可作为部分新闻源接入层 |
| `quant-crypto-watcher` | 持续重启 | 采集主链不可用 |
| Watcher 报错 | `No module named collectors.crypto_watcher.main` | 已定位为启动包缺失/错误 |
| `__init__.py`、`main.py` | 当前是目录，不是 Python 文件 | 必须修复 |
| `F:\quant-system\docker-compose*.yml` | 容器标签仍引用，但宿主机文件已不存在 | 必须先恢复可重建配置 |

结论：现有机器有可复用的存储和任务基础设施，但采集主进程、可重建配置和运行验收尚未成立。

## 三、目标架构

### 1. 数据平面

建议保留 `F:\quant-system` 作为持续采集与存储服务，承担：

- 数据源连接与轮询。
- WebSocket 断线重连。
- Redis 队列与短期去重。
- TimescaleDB 标准化时序存储。
- InfluxDB 运行指标、延迟和健康度监控。
- 数据补采与失败重试。

### 2. 研究平面

`F:\financial-alert-system` 继续作为研究系统，承担：

- 读取标准化快照和历史窗口。
- 宏观传播路径推导。
- 微观状态与证据分析。
- 数据可信度、新鲜度和冲突展示。
- 生成研究报告、验收产物与 Obsidian 摘要。

研究平面不直接控制采集容器，也不持有交易权限。

### 3. 控制平面

Obsidian 项目控制台承担：

- 阶段计划、验收记录和故障复盘。
- 数据源清单与可信度登记。
- 运行状态和门禁结论写回。

Harness 承担：

- 采集器单元测试。
- 数据契约测试。
- 去重、时间戳、异常输入和断线恢复测试。
- 研究系统读取接口的回归检查。

Harness 不承担 24 小时真实环境观察；连续运行门禁由现场观测报告完成。

## 四、统一数据契约

每条采集记录至少包含：

| 字段 | 说明 |
|---|---|
| `source` | 数据来源或交易所 |
| `dataset` | 数据集类型，如 price、funding、open_interest、macro_release |
| `symbol` | 标的；宏观数据可为空 |
| `source_timestamp` | 数据源产生时间 |
| `ingested_at` | 系统采集时间 |
| `received_at` | 系统接收时间，可用于计算网络延迟 |
| `value` / `payload` | 标准值或结构化内容 |
| `unit` | 单位和口径 |
| `source_url` | 可追溯来源 |
| `raw_hash` | 原始记录指纹，用于去重和审计 |
| `quality_status` | valid、stale、conflict、missing、invalid |
| `freshness_seconds` | 数据新鲜度 |
| `missing_reason` | 缺失或失败原因 |
| `schema_version` | 数据契约版本 |

幂等键建议：

> `source + dataset + symbol + source_timestamp + raw_hash`

必须保留 UTC 时间作为存储基准；界面层再转换为北京时间。

## 五、初始采集范围与频率

以下是第一版建议值，正式实施时根据数据源限制、成本和稳定性校准。

### 微观数据

| 数据 | 第一阶段标的 | 建议频率 | 优先级 |
|---|---|---:|---|
| 现货 / 标记价格 | BTC、ETH、SOL | 15–30 秒 | P0 |
| 永续合约价格 | BTC、ETH、SOL | 15–30 秒 | P0 |
| 资金费率 | BTC、ETH、SOL | 5 分钟及结算点 | P0 |
| 持仓量 OI | BTC、ETH、SOL | 1 分钟 | P0 |
| 期现基差 | BTC、ETH、SOL | 1 分钟 | P0 |
| 清算数据 | BTC、ETH、SOL | WebSocket 实时或 15 秒聚合 | P1 |
| 成交量与波动率 | BTC、ETH、SOL | 1 分钟 | P1 |
| 稳定币供应/流动性 | USDT、USDC | 5–15 分钟 | P1 |
| 情绪指标 | 市场级 | 1 小时 | P2 |

首期数据源采用至少两个来源交叉验证；单一来源故障时应降级，不直接填补为“正常”。

### 宏观与事件数据

| 数据 | 建议策略 | 优先级 |
|---|---|---|
| FRED 系列 | 常态低频轮询，发布窗口提高频率，使用版本/时间戳检测变化 | P0 |
| BLS 非农与通胀 | 按发布日历触发；发布窗口短周期重试并保存原始公告 | P0 |
| 美国财政部数据 | 工作日 30–60 分钟或按更新频率检查 | P1 |
| 央行公告 / 讲话 | RSS 5–10 分钟轮询，正文异步抓取 | P1 |
| 主流新闻 RSS | 5–10 分钟轮询，URL 与正文指纹去重 | P1 |
| 其他低频宏观数据 | 按官方发布日历采集，不做无意义秒级轮询 | P2 |

## 六、阶段执行计划

## S0：恢复可重建基线

状态：`待执行`

目标：停止采集器重启风暴，恢复完整、可重建的 Compose 与代码基线。

工作项：

1. 备份当前 `F:\quant-system` 和运行中容器配置。
2. 确认现有容器镜像、网络、卷、环境变量名称和依赖关系；凭据值不写入文档。
3. 恢复 `docker-compose.yml` 与本地覆盖文件。
4. 将错误目录 `collectors/crypto_watcher/__init__.py`、`main.py` 修复为真实文件。
5. 恢复 Python 包结构、启动入口和依赖锁定。
6. 加入健康检查、结构化日志和合理的重启退避。
7. 暂不接入新的数据源，只确保空载服务稳定运行。

验收门禁：

- Compose 配置可以解析并重建。
- Watcher 连续运行 30 分钟，无重启循环。
- Redis、TimescaleDB、InfluxDB 健康检查通过。
- 容器日志不出现模块缺失和凭据泄露。
- 重启 Docker 后服务可自动恢复。

## S1：建立存储与数据契约

状态：`待 S0`

目标：先建立可审计的数据落地标准，再扩大采集范围。

工作项：

1. 创建原始事件表、标准化时序表、采集运行表和数据质量表。
2. 建立唯一键与去重约束。
3. 建立 UTC 时间规范、版本字段和原始记录指纹。
4. Redis 只保存队列、短期状态和去重窗口；长期事实进入 TimescaleDB。
5. InfluxDB 保存采集延迟、成功率、重试次数、队列长度和容器心跳。
6. 提供迁移脚本、回滚脚本和最小查询样例。

验收门禁：

- 相同样本重复写入不会产生重复记录。
- 错误时间戳、空数据和异常单位被拒绝或隔离。
- 原始数据和标准化数据可以互相追溯。
- 数据库迁移可重复执行并可回滚。

## S2：上线微观 7×24 采集

状态：`待 S1`

目标：先完成 BTC、ETH、SOL 的 P0 数据连续采集。

工作项：

1. 建立交易所适配器统一接口。
2. 接入价格、资金费率、OI 和期现基差。
3. 加入限流、指数退避、熔断和备用来源切换。
4. WebSocket 断线后自动重连并补采缺口。
5. 记录来源延迟、最后成功时间和连续失败次数。
6. 将采集状态输出为只读健康接口。

验收门禁：

- 连续运行 24 小时。
- 进程重启后自动续采。
- P0 数据重复率为 0。
- 数据缺口处于预设阈值内；超阈值必须产生告警和缺失记录。
- 单一数据源中断时系统降级，研究界面显示来源异常。

## S3：上线宏观与新闻采集

状态：`待 S1，可与 S2 后半段并行`

目标：形成发布日历驱动的宏观数据和新闻事件证据链。

工作项：

1. 复用现有 BLS、FRED、财政部和 RSS 抓取逻辑。
2. 建立发布日历与窗口化调度。
3. 支持 ETag、Last-Modified 或内容哈希，减少无效请求。
4. 保存原始公告、正文、发布时间、采集时间和来源 URL。
5. 新闻正文、标题和 URL 多层去重。
6. 失败时保留原因，不以旧值伪装最新值。

验收门禁：

- 模拟发布窗口能够提高频率并在窗口结束后恢复常态。
- BLS/FRED 样本可以从原始公告追溯到标准化记录。
- 相同新闻重复抓取不会产生重复事件。
- 来源异常时研究层得到 stale/missing 状态。

## S4：质量监控、补采与运维

状态：`待 S2/S3`

目标：让“是否正在可靠采集”可以被观察和验证。

工作项：

1. 建立采集心跳、延迟、成功率、错误率、缺口和重复率指标。
2. 建立失败队列和人工可控的补采任务。
3. 设置磁盘容量、数据库连接、队列积压和数据过期告警。
4. 建立每日健康摘要与每周数据质量报告。
5. 建立容器重启、数据库恢复和令牌轮换说明。
6. 将关键状态写回 Obsidian 项目控制台。

验收门禁：

- 断网、限流、数据库短暂不可用和容器重启场景均有演练记录。
- 告警包含来源、数据集、开始时间、影响范围和恢复时间。
- 补采过程幂等，不制造重复数据。

## S5：接入研究系统

状态：`待 S2/S3`

目标：让宏观层与微观层读取标准化数据，而不是各自临时抓取。

工作项：

1. 提供只读快照与历史窗口接口。
2. 输出 `source_trust`、`freshness`、`conflict` 和 `missing_reason`。
3. 微观层优先使用采集层数据，保留浏览器即时抓取作为受控降级路径。
4. 宏观路径证据引用原始公告和标准化记录。
5. 界面展示最后更新时间、来源数量、异常来源和数据质量状态。
6. 不改变现有自动交易禁用状态。

验收门禁：

- 数据正常、过期、冲突和完全缺失四种状态均可复现。
- 研究层不会把过期数据标记为实时数据。
- 全部来源失效时输出 abstain/证据不足，而不是强制给出方向判断。

## S6：连续运行与发布门禁

状态：`待 S5`

目标：完成从功能可用到研究级运行验收的分离。

验收分三层：

1. **功能门禁**：单元测试、契约测试和回归检查通过。
2. **24 小时运行门禁**：P0 微观数据连续采集，重启恢复与缺口指标达标。
3. **72 小时稳定性观察**：无重启风暴、无持续积压、无不可解释缺口、存储增长可控。

研究级 RC 仍需：

- 真实来源稳定性证据。
- 至少一个完整观察窗口。
- 人工研究复核。
- 数据质量报告签核。

## 八、Harness 质量流水线扩展

在现有 `research-quality` 流水线中增加以下检查，但不加入交易动作：

1. Compose 配置解析与启动入口检查。
2. 数据契约 Schema 校验。
3. 去重与幂等测试。
4. UTC 时间戳与新鲜度计算测试。
5. 数据源超时、限流和异常响应测试。
6. 断线重连与检查点恢复测试。
7. 研究系统 stale/conflict/missing/abstain 回归测试。

长时间连续运行观察保留在外部运行门禁，不强塞进短周期 CI。

## 九、风险与控制

| 风险 | 控制措施 |
|---|---|
| API 限流或封禁 | 分级频率、指数退避、缓存、备用来源 |
| 数据源口径冲突 | 保留原始值、来源隔离、冲突标记，不静默平均 |
| 数据重复 | 唯一键、原始哈希、幂等写入 |
| 时间错位 | UTC 存储、源时间/接收时间/采集时间分离 |
| 容器重启风暴 | 健康检查、退避、错误上限和人工熔断 |
| 磁盘持续增长 | 保留策略、压缩、归档和容量告警 |
| 凭据泄露 | Windows/Docker Secret 或环境注入，禁止写入代码和日志 |
| 采集故障影响研究 | freshness 与 missing_reason 强制进入研究输入 |
| 范围滑向自动交易 | 继续保持 order submission、execution、deployment 禁用 |

## 十、回滚原则

- S0 修改前保存 Compose、容器检查结果和目录快照。
- 数据库变更必须通过版本化迁移执行。
- 新采集器以独立服务上线，不覆盖现有研究页面。
- 研究层接入采用可切换读路径，可回退到当前静态/即时抓取模式。
- 发生重复写入、时间错位或数据污染时，立即暂停对应来源，不扩大影响。
- 不删除原始证据；错误标准化结果通过版本和状态撤销。
