## 一、整改结论

项目暂停横向开发，停止以模块数量、测试数量和文档数量衡量进展。

未来唯一主线：

> 以 NFP 为首个试点，完成「事件前判断冻结 → 事件结果采集 → 市场反应核验 → 判断结算」的真实闭环。

在 NFP 闭环通过前，CPI、FOMC、人物库、沃什监控、微观扩展、推送、云端和新工作台功能均不得并行推进。

预计整改周期：工程收口约 1–2 周，研究验证约 6–8 周。真实事件观察时间不得用代码、历史回填或文档口径压缩。

### 阶段 3：只建设一个 NFP 垂直闭环

时间：5–7 天。

只实现以下流程：

#### 事件前

- 建立 NFP 事件；
- 设置严格 `as_of` 截止时间；
- 记录市场共识和当时可用历史数据；
- 选择不超过 10 条重点传播路径；
- 对美元、两年期美债、纳斯达克、黄金和 BTC 形成概率判断；
- 列出支持、反驳和缺失证据；
- 冻结事前报告，禁止事后修改。

#### 事件后

- 写入实际公布值和后续修正值；
- 采集规定窗口内的市场反应；
- 将事前判断与实际结果逐项比较；
- 每项结算为命中、部分命中、失败或无法判定；
- 记录失败原因和模型调整，但不改写原始预测。

#### 用户入口

只保留一个入口，回答五个问题：

1. 发生了什么？
2. 事前判断是什么？
3. 当时依据是什么？
4. 结果哪里对、哪里错？
5. 下一次应调整什么？

门禁：

- 全流程不修改代码；
- 操作不依赖开发者解释；
- 10 分钟内完成事件卡生成和冻结；
- 所有数据带来源、时间和新鲜度；
- 缺失或过期数据必须停止确认，不能显示为实时；
- 事前报告不可被事后数据污染。

### 阶段 4：历史盲测与真实事件验证

时间：6–8 周。

#### 历史盲测

最低 20 个历史 NFP 事件。

要求：

- 严格恢复当时可用信息；
- 禁止使用后续修正值和事后行情；
- 每次先冻结预测，再解锁实际结果；
- 与市场共识、政策和趋势不变、仅按数据意外方向判断以及不使用传播图的朴素模型比较。

记录：

- 方向准确率；
- 概率校准；
- Brier Score；
- 相对基线增量；
- 弃权率；
- 错误类型；
- 每次运行和维护时间。

#### 真实前瞻事件

至少完成 3 次真实事件闭环，可来自 NFP、CPI 或 FOMC，但必须使用同一套冻结和结算规则。真实事件不得用历史回填替代。

门禁：

- 历史盲测不少于 20 次；
- 真实闭环不少于 3 次；
- 数据穿越为 0；
- 所有预测 100% 按期结算；
- 关键数据过期误标实时为 0；
- 相比简单基线出现可复核的正增量；
- 每次运行不需要修改产品代码。

## 七、停止条件

出现以下任一情况，停止继续产品化：

- 无法可靠恢复历史 `as_of` 数据；
- 完成 20 次盲测后不优于简单基线；
- 连续 3 次真实事件没有产生增量结论；

- 工作台使用时间高于直接使用 Obsidian 模板；
- 项目再次新增与 NFP 闭环无关的主模块；
- 文档和治理工作再次超过实际研究工作。

> **宏观事件证据账本**，不再宣称具有预测或预警能力。

## 十三、2026-07-17 实施结果与剩余阻断

### 已完成并实测

- 建立工程基线提交 `559e7dc`，标签 `nfp-engineering-remediation-v2`；只提交本轮 NFP、状态治理和安全收口，未卷入 micro / 统一工作台等在制改动。
- NFP 事件卡升级为 `nfp-event-card-v2`：递归 canonical JSON + SHA-256 覆盖完整事前冻结体；嵌套篡改、重算洗白、磁盘篡改和 v1 旧卡自动升级均被拒绝。
- 浏览器与 Node 使用同一 SHA-256 语义；浏览器真实执行了冻结、完整性复核、嵌套篡改拒绝和结算链校验。
- 公开页面移除 `local_config.js`；静态服务只允许 `data/nfp_cards/<safe-id>.json`，其他 `data/` 路径保持拒绝。
- 验收判级改为 fail-closed；`file_exists` 只授予结构存在，手工标签和合成脚手架不得授予研究信用。
- 状态生成与 Obsidian 同步共用 `project-status-builder-v3`，代码库与库内 `project_status.json` 由同一对象生成。
- 已逻辑拆出 `apps/nfp-research` 与 `packages/official-macro-data`。验收结论是“可脱离原仓库运行的双包 bundle”，不是“NFP 单目录完全独立”。
- 在全新隔离副本、无旧 artifacts/data、无预装子项目依赖的条件下，`npm run gate:remediation-p3` 全部通过。

### 明确没有完成

- 真实历史盲测门禁当前为 **BLOCK：0/25 合格**；工程烟测构造的 20 个样本只证明门禁可达和篡改会阻断，不是研究证据。
- 真实前瞻事件仍为 **OBSERVING 0/3**；不得用历史卡、事后回填或合成行情替代。
- 当前本地哈希只能证明文件内部一致性，不能单独证明外部共识和行情确实在当时采集；正式研究信用仍需可信来源、采集时间和原始快照证据链。
- `crypto-micro` 未拆分。其现有依赖和未提交改动尚未形成干净边界，现在建目录只会制造装饰性拆分。
- 主工作树仍含其他在制改动，因此“工程基线可复现”不等于“整个项目可发布”；release / cloud 继续 **BLOCK**。

### 唯一下一步

停止继续加模块。只采集并冻结真实 Street 共识、官方发布和五资产规定窗口行情，形成不少于 20 个可审计历史盲测；同时按日历完成 3 次真实前瞻闭环。达不到相对简单基线的可复核正增量，则按本计划降级为“宏观事件证据账本”。
