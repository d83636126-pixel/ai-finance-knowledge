## 一、整改结论

项目暂停横向开发，停止以模块数量、测试数量和文档数量衡量进展。

未来唯一主线：

> 以 NFP 为首个试点，完成「事件前判断冻结 → 事件结果采集 → 市场反应核验 → 判断结算」的真实闭环。

在 NFP 闭环通过前，CPI、FOMC、人物库、沃什监控、微观扩展、推送、云端和新工作台功能均不得并行推进。

预计整改周期：工程收口约 1–2 周，研究验证约 6–8 周。真实事件观察时间不得用代码、历史回填或文档口径压缩。

## 二、整改基线

当前已确认：

- 统一研究工作台的研究级验收、发布和上云门禁均为 **blocked**；
- 代码工作树存在 33 个已修改项和 24 个未跟踪项，尚不是可发布基线；
- 旧 AUTO ACCEPTED 主要依赖 `file_exists`，不等于真实验收；
- 统一工作台存在敏感路径暴露、真实新闻未接入、stale 误标 live、跨层伪透传和验收假通过等阻断；
- 微观严格校准预览中 60 条记录保留为严格通过的数量为 0；
- P3 尚处于只观察早期，不能证明因子或交易优势；
- 历史盲测、严格 as-of 隔离和与简单基线对比尚未形成完整研究证据。

## 三、立即冻结的范围

### 停止开发

- 人物需求轴产品化；
- 沃什五大工作组自动监控；
- 新人物、新图谱节点和新资产；
- 新微观因子、卡片和页面；
- Alert、Inbox 和新闻分类扩展；
- 推送、Cron、7×24、云端和多用户；
- LLM 人物模拟和报告美化；
- 新一轮统一工作台 UI 重构。

### 保持现状但不扩展

- 微观 P3 继续只观察，不修改正式校准簿；
- 现有图谱保留，不新增节点和边；
- 现有页面只修安全和阻断性错误；
- 已有报告和历史证据保留，不删除、不回填结论。

## 四、整改阶段

### 阶段 1：确认冻结与单一真相

时间：1–2 天。

> [!note]
> 复用已有 RG0 基线，不重新建造一套报告。

执行：

1. 复核当前代码、分支、未提交改动和运行工件；
2. 将所有改动分为核心闭环、安全整改、冻结保留、临时产物和来源不明五类；
3. 不整仓暂存，不删除用户已有改动；
4. 确定唯一代码根目录和唯一当前状态源；
5. 停止所有仅凭文件存在而生成的 ACCEPTED；
6. 停止新增阶段报告和重复状态文件。

门禁：

- 每项未提交改动都有明确归属；
- 旧结论未被伪造或覆盖；
- 当前状态统一登记为「整改中，未发布，研究有效性未证明」。

### 阶段 2：安全与工程可信度收口

时间：2–4 天。

执行：

1. 阻断通过 HTTP 访问 `.env`、`.git`、本地配置、日志、源码运维目录和 `node_modules`；
2. 浏览器端不再读取任何密钥；
3. 建立真实可执行的回退路径，并完成临时目录演练；
4. 将验收注册表从「文件存在」改为「实际命令结果 + 工件哈希」；
5. 合并项目状态：一个机器状态文件，一个由机器状态生成的人类进度页；
6. 形成干净、可复现、可回滚的 Git 基线。

门禁：

- 敏感路径全部返回 403 或 404；
- 合法页面和必要数据正常访问；
- 工程测试从稳定提交运行；
- 回退演练通过；
- Git 状态清晰；
- 未通过时禁止进入功能整改。

### 阶段 3：只建设一个 NFP 垂直闭环

时间：5–7 天。

只实现以下流程：

#### 事件前

- 建立 NFP 事件；
- 设置严格 `as_of` 截止时间；
- 记录市场共识和当时可用历史数据；
- 选择不超过 10 条重点传播路径；
- 对美元、两年期美债、纳斯达克、黄金和 BTC 形成概率判断；
- 列出支持、反驳和缺失证据；
- 冻结事前报告，禁止事后修改。

#### 事件后

- 写入实际公布值和后续修正值；
- 采集规定窗口内的市场反应；
- 将事前判断与实际结果逐项比较；
- 每项结算为命中、部分命中、失败或无法判定；
- 记录失败原因和模型调整，但不改写原始预测。

#### 用户入口

只保留一个入口，回答五个问题：

1. 发生了什么？
2. 事前判断是什么？
3. 当时依据是什么？
4. 结果哪里对、哪里错？
5. 下一次应调整什么？

门禁：

- 全流程不修改代码；
- 操作不依赖开发者解释；
- 10 分钟内完成事件卡生成和冻结；
- 所有数据带来源、时间和新鲜度；
- 缺失或过期数据必须停止确认，不能显示为实时；
- 事前报告不可被事后数据污染。

### 阶段 4：历史盲测与真实事件验证

时间：6–8 周。

#### 历史盲测

最低 20 个历史 NFP 事件。

要求：

- 严格恢复当时可用信息；
- 禁止使用后续修正值和事后行情；
- 每次先冻结预测，再解锁实际结果；
- 与市场共识、政策和趋势不变、仅按数据意外方向判断以及不使用传播图的朴素模型比较。

记录：

- 方向准确率；
- 概率校准；
- Brier Score；
- 相对基线增量；
- 弃权率；
- 错误类型；
- 每次运行和维护时间。

#### 真实前瞻事件

至少完成 3 次真实事件闭环，可来自 NFP、CPI 或 FOMC，但必须使用同一套冻结和结算规则。真实事件不得用历史回填替代。

门禁：

- 历史盲测不少于 20 次；
- 真实闭环不少于 3 次；
- 数据穿越为 0；
- 所有预测 100% 按期结算；
- 关键数据过期误标实时为 0；
- 相比简单基线出现可复核的正增量；
- 每次运行不需要修改产品代码。

## 五、模块处置

| 模块 | 处置 |
|---|---|
| NFP 事前—事后闭环 | 唯一功能开发主线 |
| 核心传播路径 | 收缩至不超过 10 条 |
| BLS 及必要市场数据 | 保留并提高可信度 |
| Obsidian 事件账本 | 保留 |
| 微观 P3 | 仅观察，不扩展 |
| CPI / FOMC | 等 NFP 闭环通过后复用 |
| 统一工作台 | 安全修复后冻结 UI |
| 人物需求轴 | 研究专题，停止产品化 |
| 沃什监控 | 冻结 |
| Alert / Inbox | 保留现状，不扩展 |
| 自建实时新闻平台 | 暂停 |
| 推送、云端、7×24 | 取消当前排期 |
| 自动交易 | 永久不在本阶段范围内 |

## 六、文档治理整改

整改后只允许五类核心文档：

1. 项目目标与边界；
2. 当前状态；
3. 唯一执行清单；
4. 事件预测与结算账本；
5. 决策记录。

规则：

- 不再「一阶段一份计划、一轮一份报告」；
- 同一事实不在多个文件手工维护；
- 机器状态必须由真实测试生成；
- 文档不得自动把「存在」升级为「通过」；
- 历史报告进入归档，不继续承担当前状态；
- 没有真实事件结果，不允许使用「完成研究闭环」。

## 七、停止条件

出现以下任一情况，停止继续产品化：

- 无法可靠恢复历史 `as_of` 数据；
- 完成 20 次盲测后不优于简单基线；
- 连续 3 次真实事件没有产生增量结论；
- 每次事件都需要修改代码才能运行；
- 数据源维护成本超过研究价值；
- 工作台使用时间高于直接使用 Obsidian 模板；
- 项目再次新增与 NFP 闭环无关的主模块；
- 文档和治理工作再次超过实际研究工作。

若研究流程有用但预测不优于基线，项目降级为：

> **宏观事件证据账本**，不再宣称具有预测或预警能力。

## 八、最终验收

项目只有同时满足以下条件，才算做出第一个实际功能：

- 一个稳定、可回滚的代码版本；
- 一个安全的本地入口；
- 一个不依赖开发者的 NFP 完整流程；
- 至少 20 次历史盲测；
- 至少 3 次真实前瞻事件；
- 事前判断不可修改；
- 事后结果全部结算；
- 与简单基线完成对比；
- 能证明系统减少误判、节省时间或提供可复核增量。

在此之前，项目统一状态为：

> **工程原型存在，实际研究价值尚未通过验收。**

## 九、下一步

按顺序继续（用户已授权无需再确认）：

1. 用真实窗口行情替换合成 reactions；
2. 将 Street 共识写入冻结卡（替换 naive prior-print）；

4. 与简单基线对比并登记是否触发停止条件。
