# 远程采集面 · 详细操作步骤（照着做）

> 目标：抓取在网上跑 → 写入 Supabase → 你的电脑只读库，不再每 5 分钟本地抓。  
> 代码根：`F:\financial-alert-system`  
> 决策：[[远程采集面_Supabase_2026-07-17]]  
> 本机每天打开：[[日开会话_DayOpen_2026-07-17]]

---

## 总览（你会完成什么）

```text
第 1 步  注册并新建 Supabase 项目
第 2 步  在网页里执行建表 SQL
第 3 步  复制 URL / anon / service_role 三把钥匙
第 4 步  本机 .env 只配 URL + anon，试拉库（或用日开会话自动写）
第 5 步  选一个「非本机」定时写库方式（推荐 GitHub Actions）
第 6 步  （可选）Lovable 只读面板
第 7 步  确认本机没有 5 分钟抓取任务
第 8 步  每天：Obsidian day_open / 日开会话脚本
```

预计时间：有账号约 **30–60 分钟**（不含买 VPS）。

## 第 1 步：新建 Supabase 项目

1. 打开浏览器访问：<https://supabase.com>  
2. 用 GitHub / Google 登录。  
3. 点 **New project**：
   - **Name**：例如 `fas-remote-collector`
   - **Database Password**：自己设一个强密码，**存到密码管理器**（以后改库结构可能用到）
   - **Region**：选离你近的（如 `Northeast Asia (Tokyo)` / `Southeast Asia`）
4. 等项目创建完成（约 1–2 分钟）。

---

## 第 2 步：执行建表 SQL

1. 左侧菜单打开 **SQL Editor** → **New query**。  
2. 用记事本打开本机文件：

   `F:\financial-alert-system\supabase\migrations\20260717_remote_collector.sql`

3. **全选复制** → 粘贴到 Supabase SQL Editor。  
4. 点 **Run**（或 Ctrl+Enter）。  
5. 下方应显示 success / 无红色报错。  
6. 左侧 **Table Editor** 应能看到这 5 张表：

| 表名 | 用途 |
|---|---|
| `collector_runs` | 每次采集任务记录 |
| `market_snapshots` | 宏观/行情快照 |
| `news_items` | 新闻 |
| `micro_observations` | 微观观察 |
| `exchange_availability` | 交易所可用性 |

若报错：把完整错误信息发给助手；常见原因是重复执行（多数语句带 `if not exists`，可再跑一次）。

---

## 第 3 步：复制 API 密钥

1. 左侧 **Project Settings**（齿轮）→ **API**。  
2. 记下三样（不要发到公开群）：

| 名称 | 界面位置 | 用途 |
|---|---|---|
| **Project URL** | `Project URL` | 形如 `https://abcdefgh.supabase.co` |
| **anon public** | `Project API keys` → `anon` `public` | 本机 / Lovable **只读** |
| **service_role** | `Project API keys` → `service_role` `secret` | **仅远程写库**；点 Reveal 才显示 |

> [!danger]
> `service_role` 拥有绕过 RLS 的写权限。  
> **禁止**放进：浏览器前端、Lovable 前端、Git 仓库、聊天截图。

## 第 4 步：本机只读配置 + 试拉

### 4.1 编辑 `.env`

1. 打开：`F:\financial-alert-system\.env`  
   （若没有，复制 `.env.example` 改名为 `.env`）  
2. 增加（替换成你的真实值）：

```env
SUPABASE_URL=https://你的项目.supabase.co
SUPABASE_ANON_KEY=eyJ你的anon钥匙...
```

3. **本机不要写** `SUPABASE_SERVICE_ROLE_KEY`（除非临时做 bootstrap 且用完立刻删）。

### 4.2 试拉

在 PowerShell：

```bat
cd /d F:\financial-alert-system
npm run supabase:pull
```

**刚建库、还没写入时**，结果大致是：

```json
{ "ok": true, "counts": { "runs": 0, "market_snapshots": 0, ... } }
```

这表示**连接成功**。  
若 `NOT_CONFIGURED`：检查 `.env` 变量名拼写、是否保存、是否在该目录执行。  
若 `supabase_http_401`：anon key 或 URL 错了。

拉成功后缓存文件：

`F:\financial-alert-system\data\remote_cache\latest_bundle.json`

## 第 5 步：非本机定时写库（三选一）

抓取必须跑在**另一台环境**。按省事程度：

### 方案 A（推荐起步）：GitHub Actions 定时

适合：你已有 GitHub 仓库、不想买服务器。

#### A1. 在 GitHub 仓库加 Secrets

仓库 → **Settings** → **Secrets and variables** → **Actions** → **New repository secret**，添加：

| Secret 名 | 值 |
|---|---|
| `SUPABASE_URL` | 项目 URL |
| `SUPABASE_SERVICE_ROLE_KEY` | service_role |
| `SUPABASE_ANON_KEY` | anon（可选，用于跑后自检） |

#### A2. 加工作流文件

在仓库创建（或让助手创建）：

`.github/workflows/remote-collector.yml`

示例（每 15 分钟；可改成 `*/5` 但注意免费额度）：

```yaml
name: fas-remote-collector
on:
  schedule:
    - cron: "*/15 * * * *"   # UTC 每 15 分钟
  workflow_dispatch:          # 允许手动点一次

jobs:
  collect-and-push:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "20"
      - name: Ping Supabase writer
        env:
          SUPABASE_URL: ${{ secrets.SUPABASE_URL }}
          SUPABASE_SERVICE_ROLE_KEY: ${{ secrets.SUPABASE_SERVICE_ROLE_KEY }}
        run: node scripts/remote_collector_push.js --ping
```

> 当前仓库的 `remote_collector_push.js --ping` 先验证「远程能写库」。  
> 下一步再把真正的 `collect:micro:once` + market refresh 串进同一 Job（需要交易所出网；Actions 默认可出网）。

#### A3. 验证

1. Actions → 选该 workflow → **Run workflow**。  
2. 成功后，回 Supabase **Table Editor** → `collector_runs` 应出现新行。  
3. 本机再执行：`npm run supabase:pull`，`counts.runs` 应 > 0。

---

### 方案 B：便宜 VPS / 云主机

适合：要每 5 分钟稳定跑、逻辑更重。

1. 买一台小 Linux（1 核即可）。  
2. 安装 Node 20，把仓库 clone 上去。  
3. 在 VPS 的 `.env` 写入：

```env
SUPABASE_URL=...
SUPABASE_SERVICE_ROLE_KEY=...
FRED_API_KEY=...   # 若要刷宏观
```

4. crontab 示例（每 5 分钟）：

```cron
*/5 * * * * cd /opt/financial-alert-system && /usr/bin/node scripts/run_micro_continuous_collect.js --once >> /var/log/fas-collect.log 2>&1 && /usr/bin/node scripts/remote_collector_push.js --from-local-artifacts >> /var/log/fas-push.log 2>&1
```

5. **不要**在这台 VPS 上开图形桌面长期挂浏览器。

---

### 方案 C：本机一次性 bootstrap（仅验收写通路，不是长期方案）

若你想先确认「能写进库」，可在本机临时：

```bat
cd /d F:\financial-alert-system
```

临时把 `SUPABASE_SERVICE_ROLE_KEY` 写进 `.env`，然后：

```bat
npm run supabase:push:ping
```

成功后：

1. 立刻从 `.env` **删掉** `SERVICE_ROLE`  
2. 仍以方案 A/B 做长期定时  
3. **不要**重新安装本机 5 分钟计划任务

## 常见问题

| 现象 | 处理 |
|---|---|
| pull 报 NOT_CONFIGURED | `.env` 未配或没在项目根目录执行 |
| 401 / JWT | URL 或 key 复制错、多了空格 |
| 表是空的 | 正常：还没远程 push；先 `push:ping` |
| Actions 失败 | 检查 Secrets 名称是否完全一致 |
| 电脑又卡了 | 查是否重新安装了本机 schedule |
