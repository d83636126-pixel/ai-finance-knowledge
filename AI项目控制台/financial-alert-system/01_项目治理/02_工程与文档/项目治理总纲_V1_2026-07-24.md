# financial-alert-system 项目治理总纲 V1

## 1. 治理结论

项目已结束“架构重做”阶段，进入“产品运行与证据积累”阶段。

后续不再把所有事项串成单一路径，也不再用研究门禁阻断产品工程。项目按治理、产品、运行、数据、研究五条轨道管理；每条轨道有独立目标和独立验收语义。

## 2. 当前权威基线

| 领域 | 当前状态 | 可以声称 | 不可以声称 |
|---|---|---|---|
| 架构 | `ARCHITECTURE_ACCEPTED` | 架构重做完成，基线为 `4bfbb066` | 已生产化 |
| 开发运行 | `RUNTIME_ACCEPTANCE_PASS` | synthetic 事件可经 CardService → API → Graph 连通 | 生产 SLA |
| 本地产品 | `LOCAL_PRODUCT_SLICE_PASS` | 本地真实 HTTP 产品切片可操作，tip 为 `3dbb66e` | 正式发布 |
| 研究证据 | `ABSTAIN_NO_UNSEEN_EVIDENCE` | 当前没有合格未见证据 | `RESEARCH_PASS`、模型有效 |
| 发布部署 | `NOT_STARTED` | 当前维持本地运行 | 已上云、可对外服务 |

以上状态相互独立。产品可继续迭代，不需要等待 `RESEARCH_PASS`；研究状态也不得因工程或产品通过而升级。

## 3. 五条治理轨道

| 轨道 | 目的 | 近期结果 | 放行标准 |
|---|---|---|---|
| `GOV` 治理 | 管理目标、优先级、风险、决策和验收语义 | 单一治理正本与30天路线 | 文档无双入口；任务有明确边界 |
| `PROD` 产品 | 形成真实日常使用流程 | 从“能打开”升级到“每天可用” | 用户流程、错误态、回滚验收通过 |
| `OPS` 运行 | 保证启动、健康检查、恢复和回归 | 稳定本地运行基线 | 启动检查、健康状态、故障恢复通过 |
| `DATA` 数据 | 管理来源、新鲜度、修订、缺失和降级 | 质量状态进入产品界面 | 数据过期或异常必须显式降级 |
| `RES` 研究 | 独立积累前瞻证据和评价模型 | live-forward 被动积累 | 只按预注册门槛评价，不反向污染开发 |

执行资源默认优先顺序：`PROD / OPS / DATA` > `RES` > 候选增强。

## 4. PASS 语义隔离

以下结论不得相互替代：

- `ARCHITECTURE_ACCEPTED`：结构和边界验收通过。
- `RUNTIME_ACCEPTANCE_PASS`：开发环境链路能够运行。
- `LOCAL_PRODUCT_SLICE_PASS`：本地产品切片可操作。
- `DATA_QUALITY_PASS`：指定数据在指定时间点满足质量策略。
- `RESEARCH_PASS`：真正未见证据满足预注册研究门槛。
- `RELEASE_PASS`：目标发布环境满足安全、运行、回滚和运维要求。

任何报告必须写明自己属于哪一种 PASS，并显式列出不包含的结论。

## 5. 单一真相源

项目只保留三类治理入口：

1. **战略正本**：本文件，负责阶段、轨道、优先级和风险。
2. **当前执行入口**：`docs/ai-collab/当前执行入口.md`，只指向当前一个闭环。
3. **历史证据**：`docs/ai-collab/闭环归档/`，只保存已经结束的闭环。

`Cursor_Codex_闭环交接板.md` 只负责一个可执行任务，不承担长期路线图。

旧架构计划、盲测计划、上云计划和功能设想均为历史或候选材料；除非被本总纲选入当前轨道，否则不得作为执行指令。

## 6. 未来30天路线

### 第一阶段：GOV-01 治理换轨

- 归档 CC-13。
- 建立本总纲及 Obsidian 导航。
- 固化五种当前状态和六类 PASS 语义。
- 建立首版风险登记和候选池。
- 当前工作树中的未归属恢复改动只登记、由 Human 决定，不自动删除或纳入基线。

完成标志：治理索引只把“当前执行入口”作为执行指针；本总纲是唯一战略正本。

### 第二阶段：PROD-01 日常运行加固

- 固化一键启动和停止方式。
- 在产品页展示服务健康、数据新鲜度和降级原因。
- 给主要用户流程增加最小回归验收。
- 建立故障恢复和日常使用说明。

完成标志：用户无需阅读工程文档即可启动、判断状态、发现故障并恢复。

### 第三阶段：OPS-01 / DATA-01

- 汇总现有关键 smoke 为分层验收入口。
- 对数据来源应用策略化新鲜度、版本和修订状态。
- 记录运行事故、恢复时间和重复故障。

完成标志：运行失败与数据过期可区分、可定位、可回放。

`RES` 轨道在上述阶段中继续被动积累 live-forward 证据，但不阻断产品主线。

## 7. 首版风险登记

| ID | 风险 | 当前处理 |
|---|---|---|
| `R-GOV-01` | 历史计划重新成为当前入口 | 索引只指向当前执行入口；旧计划标记 historical |
| `R-ENG-01` | `apps/api` 相对路径消费 `graph-domain` | 进入工程风险池，按实际故障或发布需要处理 |
| `R-ENG-02` | API 与 contracts 存在契约漂移可能 | 纳入回归和契约检查，不立即重构 |
| `R-OPS-01` | legacy suite 未完整聚合现有验收 | 在 OPS-01 建分层入口 |
| `R-UI-01` | browser UMD 需要显式 freshness 检查 | 保留现有 gate，后续并入统一验收 |
| `R-RES-01` | 没有真正未见研究证据 | 保持 `ABSTAIN_NO_UNSEEN_EVIDENCE`，不阻断产品 |
| `R-WT-01` | 工作树曾出现恢复临时文件，当前 `architecture.html` 有未归属修改 | 不触碰；Human 确认来源后决定保留、归档或恢复 |

风险只有在影响当前目标时才升级为阻断，不能因为“未来可能有用”自动进入开发。

## 8. 候选池与暂缓项

以下内容不进入近期主线：

- 个人 MCP 门面；
- 付费 WorldMonitor API 或冲突数据；
- 大规模前端重写；
- 数据库迁移或图数据库；
- 上云、多用户、自动交易；
- Historical Similarity、Alert Ranking、AI Analyst；
- 为取得 `RESEARCH_PASS` 过度投入历史样本修复。

候选项只有在存在明确用户场景、收益、成本和验收标准后，才能被提升为当前任务。

## 9. 任务开启规则

每个闭环必须包含：

- 所属轨道；
- 一句话目标；
- 明确成功标准；
- 明确不做；
- 可回滚点；
- 独立验收证据；
- 对 PASS 语义的准确声明。

默认同时只允许一个工程写租约。跨轨道事项可以并行观察，但不能出现两个“当前执行入口”。

## 10. Human 决策边界

以下事项必须由 Human 明确授权：

- 删除、历史重写、强制合并；
- 密钥轮换和外部账号操作；
- 付费数据或服务采购；
- 上云和对外发布；
- 改变研究门槛；
- 将候选项提升为正式路线；
- 处理来源不明的临时文件。

## 11. 下一项

当前只开启 `GOV-01`。GOV-01 完成并归档后，由 Human 决定是否开启 `PROD-01`。

不得自动开启 CC-14、held-out/forward 评分、`RESEARCH_PASS`、上云或大规模前端改造。
