# financial-alert-system：核心动力（分析/预判依赖什么）

## 一句话

**核心动力是「传播图谱（结构化先验）+ 验证器（事后证据）」；LLM 是表述层，不是真理来源。**

应用内知识库只是图谱的目录；Obsidian 是人的研究记忆——二者都不是运行时分析引擎。

## 依赖栈（从内到外）

```text
① 传播图谱 propagation_graph.json
   节点 / 边 / 方向 / 权重 / 时滞 / 线路
        ↓
② 路径与敞口（PropagationEngine）
   事件 → 传导路径 → 资产篮子影响
        ↓
③ 验证器 / 市场证据（9 项 verifier + snapshot）
   路径是否被数据激活、事后对照
        ↓
④ Case / 日历 / Inbox 事件输入
   触发「对哪件事做分析」
        ↓
⑤ ScenarioReasoner +（可选）DeepSeek
   把 ①②③ 组织成事前/事后报告与情景表述
        ↓
⑥ 分析结果存档 / 推送（下游）
```

## 做预判、分析时，各自依赖什么

| 产出 | 真正依赖 | 不依赖（或只辅助） |
|------|----------|-------------------|
| 传导路径 / 谁影响谁 | 图谱边与权重 | 知识库 Tab、Obsidian 正文 |
| 资产篮子敞口 | 图谱 target 分层 + 路径 | LLM 单独空想 |
| 事前报告结构 | 图谱路径 + Case/事件字段 +（可选）MC | 知识库条目本身 |
| 事后对比 | 事前报告 + actual + verifier evidence | 推送 |
| 盲测可信度 | 图谱版本 + as-of 隔离 + 历史样本 + 事后打分 | 「写得好」的文案 |
| 走向预判（V2 情景） | 同上路径逻辑 + 触发条件 + 观察窗 | 点位承诺、裸 LLM |

## 核心动力 vs 周边

| 层 | 角色 |
|----|------|
| **图谱** | 发动机：编码「宏观冲击如何传到 BTC/Nasdaq/Gold/USD/Treasury」 |
| **验证器** | 刹车与仪表：用数据检验路径是否成立 |
| **事件入口** | 油门：日历/Inbox/Alert 决定何时点火 |
| **LLM** | 仪表盘文案：把结构化结果说成人话；无 Key 时仍有结构化兜底 |
| **知识库 Tab** | 说明书目录：方便找图谱里有什么 |
| **Obsidian** | 车库手记：方法论与项目记忆，不进推理循环 |

## 因此「变强」该拧哪颗螺丝

若目标是预判/分析更可信，优先顺序应是：

1. **图谱质量**（边是否对、权重/时滞是否合理、版本可追溯）
2. **验证与盲测闭环**（evidence、as-of、命中率）
3. **事件覆盖与归类**（日历/突发能否正确挂到 graph node）
4. 然后才是文案、推送、知识库检索体验

把力气花在「知识库 Tab 变百科」或「只堆 LLM prompt」上，都没有拧到核心动力。
