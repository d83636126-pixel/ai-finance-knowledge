# 一、必须单独完成的关键路径

以下任务必须由集成负责人顺序推进。前一项未完成，不启动下一批并行开发。

## 0｜S0 当前状态确认与任务冻结

**顺序：第 0 位，所有工作之前。**

工作内容：

- 确认正式代码路径、分支和最新文件；
- 固定当前测试结果及已知缺口；
- 暂停无关功能进入微观层；
- 将本计划拆成可分派工作包；
- 明确本轮不建设真实下单适配器。

完成门禁：当前状态、范围、负责人和禁止项均有记录。

## 1｜S1 Git 基线、版本统一和公共契约冻结

**顺序：第 1 位；完成后才能启动 P1。**

工作内容：

- 把当前 `micro-layer/` 纳入 Git 基线；
- 统一代码、README、页面和状态文档的版本号；
- 冻结 `MicroSnapshot`、`SourceTrust`、`MarketState`、`FactorResult`、`CalibrationRecord`、`PortfolioContext`、`Decision`、`AuditTrail`；
- 建立健康、冲突、缺失、校准失败和组合超限等固定夹具；
- 明确核心编排文件的唯一维护人。

完成门禁：工作树可回滚、契约测试通过、固定夹具可供所有线路使用。

## 3｜S2 第一次统一集成

**顺序：第 3 位；等待 P1 各线达到最低交付标准后开始。**

工作内容：

- 合并六条线路；
- 处理公共契约漂移；
- 检查同因子重复计票；
- 检查缺失值、来源冲突和弃权逻辑；
- 验证校准、组合和页面之间的数据传递；
- 冻结第二轮强化的缺陷清单。

完成门禁：主干可运行，六条线路接口全部接通，没有阻断性集成错误。

## 5｜S3 完整验收与研究级 RC 发布

**顺序：第 5 位；等待 P2 强化完成后开始。**

工作内容：

- 重跑全部微观专项测试；
- 重跑原项目回归测试；
- 执行真实交易所探针和浏览器验收；
- 验证校准簿、组合压力和越权门禁；
- 更新 `project_status.json`、Obsidian 任务进度和验收记录；
- 形成干净 Git 基线与已知缺口清单。

完成门禁：研究级 RC 可复现、可回滚、可观察，但保持 `advisory-only`。

## 7｜S4 是否进入模拟执行评审

**顺序：第 7 位；必须等待观察期证据。**

工作内容：

- 评估 Shadow/Paper 结果；
- 检查校准漂移和失效记录；
- 审查异常处理、人工授权和杀开关；
- 决定继续研究、进入模拟执行或退回整改。

完成门禁：必须单独立项和人工授权。既有门禁不能自动升级为下单能力。

> [!warning] 2026-07-13 提前评审结果
> 已按本清单做过一次 **否决性** S4 评审（[[微观层_S4模拟执行评审_2026-07-13]]）：**不进入模拟执行**。待 S3 RC + P3 观察证据齐备后，须重新单独立项，不能沿用本次提前否决作为“已授权进入”。

# 二、可并行的工作包

## 2｜P1 六条主开发线

**顺序：第 2 位；S1 完成后六条线同时启动。**

| 并行编号 | 工作线 | 主要任务 | 独立交付物 | 不允许修改 |
|---|---|---|---|---|
| P1-A | 数据源与可信度 | Binance/Bybit/OKX、单位、新鲜度、熔断 | 交易所适配器、来源健康结果、真实探针 | 决策与组合规则 |
| P1-B | 状态与因子 | 状态节点、因子去重、代理边界 | 状态计算模块、因子目录、场景测试 | 数据适配器和校准规则 |
| P1-C | 历史校准 | 滚动样本外、非重叠事件、失效条件 | 校准管线、校准簿、研究摘要 | 页面和执行门禁 |
| P1-D | 组合风险 | 相关性、目标波动率、滑点、风险预算 | 组合约束模块、压力测试 | 原始数据采集 |
| P1-E | 页面与可观测性 | 来源、状态、校准、组合、弃权原因展示 | 诊断页面、审计链、夹具渲染 | 状态与决策算法 |
| P1-F | 测试与状态治理 | 自动验收、回归、状态生成 | 可执行验收套件、状态同步脚本 | 业务算法 |

### P1 最低汇合条件

各线不要求全部增强完成，但进入 S2 前至少满足：

- 有明确模块输出；
- 本线测试通过；
- 缺失和失败路径有覆盖；
- 没有擅自修改公共契约；
- 证据边界和已知缺口已记录。

## 4｜P2 异常、证据和回归强化

**顺序：第 4 位；S2 集成完成后同时启动。**

| 并行编号 | 强化线 | 场景 |
|---|---|---|
| P2-A | 数据故障 | 单所失效、两所冲突、字段过期、单位不一致、熔断 |
| P2-B | 状态错误 | 重复计票、代理冒充事实、缺失值显示为零、方向冲突 |
| P2-C | 校准稳健性 | 样本外翻转、样本不足、记录过期、成本后失效 |
| P2-D | 组合压力 | 高相关资产同时扩仓、滑点上升、杠杆超限、波动率冲击 |
| P2-E | 页面与审计 | 每项结论可回溯、弃权原因完整、异常状态可见 |
| P2-F | 安全与回归 | LLM/force 越权、授权过期、杀开关、原项目全量回归 |

P2 完成后统一进入 S3，不允许各线路自行宣布发布完成。

## 6｜P3 Shadow/Paper 观察期

**顺序：第 6 位；S3 发布后开始。**

观察期内部可以并行进行：

| 并行编号 | 观察线 | 记录内容 |
|---|---|---|
| P3-A | 数据源稳定性 | 可用率、延迟、缺口、熔断和恢复 |
| P3-B | 状态有效性 | 状态出现频率、重复率、误报和冲突 |
| P3-C | 校准漂移 | 期望收益、命中率、样本外符号和失效日期 |
| P3-D | 组合效果 | 建议前后风险、相关性、滑点和最大不利变动 |
| P3-E | 产品可用性 | 用户能否快速理解状态、证据和弃权原因 |
| P3-F | 安全审计 | 授权、杀开关、越权尝试和审计链完整性 |

观察任务可以同时执行，但必须覆盖足够日历时间和真实事件，不能因为六条线同时运行就把 3–4 周缩短成几天。

# 三、推荐人员或代理分工

在 4 个并发执行位下，推荐：

| 执行位 | 第一批职责 | 第二批职责 |
|---|---|---|
| 1 集成负责人 | S0、S1、公共契约 | S2、S3、状态同步 |
| 2 数据负责人 | P1-A 数据可信度 | P2-A 数据故障、P3-A 观察 |
| 3 研究负责人 | P1-B + P1-C | P2-B + P2-C、P3-B + P3-C |
| 4 产品与风险负责人 | P1-D + P1-E + P1-F | P2-D/E/F、P3-D/E/F |

若只有单人开发，也按相同工作包拆分，通过独立分支、worktree 或分时任务保持边界，避免多类变更混在一次提交中。

# 四、启动纪律

1. S0、S1 未完成，不启动六线并行。
2. 并行线只交付自己的模块和测试，不直接改写其他线路规则。
3. 公共契约只能由集成负责人调整，并同步所有线路。
4. 每日设置固定合并窗口，其余时间不得随意向主干推入未验证变更。
5. S2 和 S3 是统一汇合点，各线路不得绕过汇合点自行发布。
6. 观察期只记录和复盘，不接入真实下单。

# 五、最短关键路径

> S0 状态冻结 → S1 基线与契约 → P1 六线并行 → S2 统一集成 → P2 六线强化 → S3 完整验收 → P3 观察期 → S4 模拟执行评审。

工程开发的最短关键路径约 **8–10 个工作日**；进入下一阶段决策所需的完整周期约为 **8–10 个工作日加 3–4 周观察期**。
