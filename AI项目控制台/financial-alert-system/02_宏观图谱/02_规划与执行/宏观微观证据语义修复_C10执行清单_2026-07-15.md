- 支持证据：0；
- 反向证据：0；
- 缺失：1；

同时发现两个P0问题：

2. `btc_price` 被映射到 `btc_etf_flow_verifier`，导致“BTC价格”实际使用Farside ETF净流量（M USD）作为证据，存在对象与单位错配。

因此，C11之后的界面建设必须由本清单阻断：**先修证据语义和状态传播，再继续仪表盘。**

## 1. 不可突破的边界

- 不执行 `git reset --hard`、`git checkout --`、`git clean`。
- 不覆盖当前dirty工作区中与本任务无关的修改。
- 不自动push，不使用 `git add .`。
- 不为了得到确认结论而降低证据或校准门槛。
- 不把ETF流量、代理数据、fixture、恢复快照或过期数据显示为实时价格证据。
- 保持 `advisory-only`、`executable=false`、`human_review_required=true`。
- 不增加下单、授权、目标仓位或自动交易能力。
- 本轮通过不代表24h、72h、P3 28天或研究级RC通过。

## 2. 正确状态模型

必须将以下状态分离：

```text
context_freshness       上下文文件本身是否在有效时间内
macro_evidence_status   宏观路径证据是否支持/反驳/缺失/过期
micro_evidence_status   funding/OI/基差等微观证据状态
overall_research_status 跨层综合研究状态
```

最低状态规则：

| 宏观证据 | 微观证据 | 综合研究状态 |
|---|---|---|
| 新鲜且支持 | 支持 | 可确认 |
| 过期或缺失 | 支持 | 部分支持或证据不足 |
| 支持 | 反驳 | 混合或反驳 |
| 反驳 | 支持 | 混合 |
| 任一核心证据失效 | 任意 | 不可确认 |

硬门禁：

```text
macro_evidence_status ∈ {stale, missing, insufficient, invalid}
→ overall_research_status != confirmed
→ readiness != ready
```

### C10-B：修复BTC证据语义

主要文件：

- `scripts/lib/macro_evidence_loop.js`
- `verifier_data_dictionary.json`
- `static/ingest/verifier_data_fetchers.js`
- 对应测试文件。

任务：

1. 删除错误别名：

```text
btc_price → btc_etf_flow_verifier
```

2. 新增或接入真实BTC价格验证项：

```text
id: btc_price
object: BTC现货价格
unit: USD
frequency: realtime/intraday
source: 至少两个健康公开市场来源，或明确降级为单源
```

3. ETF流量保持独立：

```text
btc_etf_flow → btc_etf_flow_verifier
object: BTC ETF净流量
unit: M USD
frequency: daily
role: auxiliary_daily
```

4. 增加对象与单位契约：价格、流量、收益率、指数不能直接互相别名。

验收：

- `btc_price.current_value` 为合理的BTC美元价格；
- `btc_price.data_source` 为价格来源；
- ETF流量只进入 `btc_etf_flow` 或作为标注清晰的辅助证据；
- 单位不一致或对象错误时测试失败；
- 不因缺少ETF流量而把BTC价格判为缺失。

### C10-D：统一FRED与来源健康状态

当前冲突：

- `market_snapshot.json` 记录 `has_fred_api_key=true`；
- `project_status.json` 记录 `has_fred_key=false`。

任务：

- 建立唯一密钥可用性检测逻辑；密钥只从安全配置或Windows凭据读取，不写入仓库。
- 区分：
  - 实时刷新成功；
  - 使用最近已知良好快照；
  - 快照过期；
  - 刷新失败；
  - 无密钥或无数据。
- 刷新失败允许继续生成研究报告，但必须降级，不能恢复为实时确认。
- `project_status.json`、`market_snapshot.json`、宏观来源报告与页面必须对来源可用性给出一致结论。

验收：

- 无FRED密钥时，FRED依赖项明确缺失/降级；
- 有密钥且刷新成功时，数据时间和来源可验证；
- 使用恢复快照时，页面显示恢复/过期状态；
- 不把历史 `has_fred_api_key` 字段当作当前可用性。

### C10-F：修复跨层页面表达

页面首屏必须分栏显示：

```text
宏观路径状态：证据过期/缺失/支持/反驳
微观证据状态：支持/反驳/不足
综合研究结论：确认/部分支持/混合/反驳/证据不足
```

必须同时展示：

- 数据发生时间；
- 最近刷新时间；
- 来源和单位；
- 支持、反证、缺失与过期；
- 最大证据缺口；
- 失效条件；
- 人工复核要求。

禁止继续出现以下组合：

```text
宏观路径证据过期
+ 微观局部支持
= 核心路径 confirmed / 最大缺口无
```

验收：

- 任何 `confirmed` 均可追溯到新鲜宏观与微观核心证据；
- ETF流量不会显示为BTC价格；
- fixture、恢复快照和过期来源有醒目标记；
- 无交易、授权或目标仓位控件。
