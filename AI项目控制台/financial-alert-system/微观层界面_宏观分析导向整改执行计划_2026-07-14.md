# 微观层界面：宏观分析导向整改执行计划（2026-07-14）

## 执行结论

> **阶段一至七已完成（2026-07-14）。** 微观页改为宏观验证导向；`smoke:micro-page` 与 `accept:micro-s3:offline`（14/14）通过。浏览器端到端可按需复验。
>
> **宏观上下文只读接入（2026-07-14）：** `npm run export:micro-macro-context` 从 case report 写入 `artifacts/micro_macro_context.json`；页面「宏观推导验证」可读情景/路径/资产/失效条件。不改传播图算法。

保留数据、校准、可信度、风险约束和安全门禁能力，但在用户没有再次明确授权前，项目不向自动交易、下单适配器或实盘执行延伸。

预计工期：**5–7 个工作日**。P3 每 8 小时观察继续运行，不因界面整改中断。

## 一、整改原则

1. 不建设下单、券商适配器或自动执行。
2. 不修改正式校准簿和当前 P3 候选。
3. 不改变微观计算、可信度和弃权算法。
4. 优先调整信息层级、语言、状态展示和宏观映射。
5. 执行门禁继续存在，但降级为安全边界，不再作为产品发展主线。
6. 微观层的核心任务是验证、补充或反驳宏观推导，不独立宣称价格方向。

## 二、阶段一：定位与语言整改

工期：0.5–1 天。

| 当前内容 | 调整后 |
|---|---|
| 加密微观层 · 状态与仓位边界 | 加密微观分析 · 状态与证据 |
| 仓位建议与弃权原因 | 风险判断与证据边界 |
| 可执行建议 | 重点关注标的 |
| 校准与组合上下文 | 历史证据与风险约束 |
| P5 执行门禁 | 安全边界 |
| 建议 / 弃权 | 支持 / 反驳 / 证据不足 |

页面必须明确显示：

> 本页面用于验证宏观推导，不生成订单，不提供自动交易能力。

### 验收条件

- 首屏不再以“仓位、执行、下单”为主要语言；
- 自动交易只在安全声明中出现；
- 保留 `advisory-only` 和杀开关状态。

## 三、阶段二：重构首屏信息层级

工期：1 天。

将当前五个工程 KPI 调整为：

1. 当前微观风险状态；
2. 重点关注标的；
3. 平均证据质量；
4. 主要证据缺口；
5. P3 观察状态。

新增“当前结论”主卡片：

- 核心判断；
- 支持证据；
- 反向证据；
- 最大不确定性；
- 与宏观推导的关系；
- 更新时间。

### 验收条件

- 用户在 10 秒内能回答“当前发生了什么、证据是否可靠、应关注什么”；
- 状态节点和因子数量退到二级信息；
- `0 actionable` 不再占据首屏。

## 四、阶段三：采集进度与来源健康

工期：1 天。

将“采集中……”升级为分步骤状态：

- 正在读取 Binance、Bybit、Bitget、OKX；
- 已完成标的数；
- 当前耗时；
- 成功、降级、失败来源；
- 是否使用缓存；
- 失败是否影响最终结论。

### 状态色

| 状态 | 含义 |
|---|---|
| 绿色 | 服务及核心来源正常 |
| 黄色 | 部分来源降级，但满足最低覆盖 |
| 红色 | 核心数据不足，结论不可用 |
| 灰色 | 使用缓存或尚未采集 |

Farside、OKX 等错误必须在页面显示具体影响，不能只出现在控制台。

### 验收条件

- 加载过程中不再整页显示无解释的 0；
- 超过 10 秒显示预计等待说明；
- 单个非核心来源失败不会让整个页面看起来崩溃；
- 浏览器错误和页面来源状态一致。

## 五、阶段四：建立宏观—微观映射

工期：1–2 天。

新增“宏观推导验证”区域：

| 内容 | 示例 |
|---|---|
| 当前宏观情景 | 美元流动性偏紧 |
| 图谱核心路径 | 实际利率 → 风险偏好 → BTC |
| 微观证据 | funding 拥挤、OI 变化、跨所分歧 |
| 判断 | 支持 / 部分支持 / 反驳 / 尚无证据 |
| 影响资产 | BTC、ETH、SOL |
| 失效条件 | funding 回落、OI 下降、风险状态反转 |

### 接入原则

- 只读取传播图或共享状态，不修改传播图算法；
- 微观层只验证路径，不替代宏观结论；
- 没有宏观上下文时明确显示“当前为独立微观观察”。

### 验收条件

- 每项微观结论能说明其对应或反驳哪条宏观路径；
- 不把相关性写成因果事实；
- 支持、反驳和证据不足三种结果均有测试。

## 六、阶段五：增加趋势与 P3 观察面板

工期：1–2 天。

增加：

- funding 24h / 7d 趋势；
- OI 及跨所分歧趋势；
- 数据质量变化；
- 市场状态变化；
- P3 候选匹配记录；
- 弃权原因变化；
- 来源可用率。

### P3 状态条

- 观察候选；
- 已运行天数；
- 剩余天数；
- 观察次数；
- 候选匹配次数；
- 异常数量；
- 正式校准簿是否未变化；
- actionable 是否保持为 0。

### 验收条件

- P3 状态与 `micro_p3_status.json` 一致；
- 每 8 小时观察日志可以形成时间序列；
- 页面不提供“启动执行”或“授权下单”按钮。

## 七、阶段六：路线图与帮助信息整改

工期：0.5 天。

底部路线调整为：

> 数据可信度 → 状态解释 → 宏观映射 → 历史验证 → 持续观察

原有组合与执行内容处理：

- 组合约束改为“风险约束”；
- 执行门禁收进“安全边界”折叠区；
- 删除任何暗示未来必然进入自动交易的描述；
- S4 改成“研究阶段复评”，不默认指向模拟执行。

## 八、阶段七：完整验收

工期：1 天。

### 功能验收

- 7 个标的正常加载；
- 缺失值仍显示 `--`；
- 来源降级可见；
- 宏观上下文有 / 无两种模式正常；
- P3 状态读取正确；
- 支持、反驳、证据不足显示正确；
- 正式校准簿不被修改。

### 安全验收

- `advisory-only` 保持开启；
- 杀开关保持开启；
- actionable 保持 0；
- 无下单入口；
- 无自动授权逻辑；
- P3 自动观察正常运行。

### 回归验收

- 微观专项测试；
- P3 专项测试；
- 浏览器端到端；
- 项目 `check`、`smoke`、`acceptance`；
- 首页、传播图、架构页不受影响。

## 九、实施顺序

```text
定位与语言
    ↓
首屏重构
    ↓
采集进度与来源健康
    ↓
宏观—微观映射
    ↓
趋势与 P3 观察
    ↓
路线与帮助整改
    ↓
完整验收
```

阶段一至三必须串行；阶段四和阶段五可以并行；最终统一进入阶段六、七。

## 十、文件范围

### 主要修改

- `micro-layer/index.html`；
- `micro-layer/micro_dashboard.js`；
- 微观页面专项测试；
- P3 状态读取与展示模块；
- 必要的只读宏观上下文接口。

### 明确不修改

- 传播图核心推导算法；
- 正式校准簿；
- P3 候选和观察门槛；
- 自动执行权限；
- 真实订单能力。

## 十一、完成标准

> **微观页首先回答“宏观判断是否得到微观证据支持”，其次解释风险和不确定性，不再以仓位或执行作为页面中心。**
