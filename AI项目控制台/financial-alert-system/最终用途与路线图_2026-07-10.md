# financial-alert-system：最终用途与路线图

## 最终用途（已确认）

> **宏观传导研究 + 预警能力补强的混合系统。**
>
> - 图谱用于补全 Alert 侧「识别范围不足、缺宏观链路」的问题。
> - 核心场景：**事前分析** + **事后对比分析**。
> - **推送功能延后**，等其他能力闭环后再做。

一句话：

**先用图谱把宏观链路讲清楚、验清楚；扫描与推送是下游，不是第一优先级。**

## 产品分层（2026-07-10 状态）

| 优先级 | 内容 | 状态 |
|--------|------|------|
| **P1** | 传播图、事前 Case、事后 evidence 对比、验证数据源、日历/Inbox | ✅ **已完成** |

## 第一版最小闭环（验收）— 已达成

一次完整的 **事前 → 事后** 闭环：

1. ✅ **事前**：Case / 日历 / Inbox → 路径 + 验证清单
2. ✅ **数据**：9 项 verifier E2E 可用（含 TIPS/DXY 等）
3. ✅ **事后**：日历/Inbox「事后对比」+ 推演页
4. ✅ **Alert 桥接**：Dashboard「看传导」「入 Inbox」

## 下一阶段规划（建议顺序）

1. **回测可信度**：MAJOR_EVENTS 补历史 NFP/CPI；建立 `as_of_cutoff` 和字段隔离，禁止 actual、修正值、事后行情进入事前报告
2. **体验**：事前/事后结果常驻最小版，先完成保存、刷新保留和事件/图谱跳转
3. **工程**：代码目录独立 GitHub 仓
4. **P3**：推送（产品确认后）

## 历史分析口径（2026-07-11 补充）

- **复盘模式**：可以读取 actual 与事后行情，用于解释实际传导。
- **历史盲测模式**：以事件公布时间为信息截止点，只读取当时可得数据。
- 盲测报告必须先保存，再解锁 actual 并生成事后对比。
- 没有严格 as-of 隔离的历史分析只能称为复盘，不能宣称为无泄漏回测。
