# AI记录规范

## 什么时候需要写入 Obsidian

以下情况必须记录：

- 项目目标发生变化。
- 产生了新的关键判断。
- 完成了一个阶段性动作。
- 出现需要以后继续处理的问题。
- 用户明确表达了偏好、原则、方法。
- 发现了新的资料来源或证据。
- 产生了下一步行动。

### 明确不需要记录的情况（反例）

以下情况**不需要**写入 vault，避免噪音：

- 纯闲聊、寒暄、非工作对话。
- 临时想法、未确认方向、用户说"先想想"的内容。
- 已废弃的方向（用户已明确说"不做"或"换方案"）。
- 一次性信息查询（如"今天天气""现在几点"）。
- 纯技术故障排查过程（排查完不需要记录每一步踩坑细节，只记录根因和解决方案）。
- 用户明确说"不要记录"的任何内容。

### AI 自动判断六类触发器

AI 在会话中遇到以下六类信息时，应自动判断是否需要写入：

1. **新决定** → 写入决策记录
2. **新任务** → 写入任务进度（进行中栏）
3. **新进度** → 写入任务进度（已完成栏）
4. **新风险** → 写入任务进度（阻塞栏）
5. **新资料** → 写入资料索引
6. **下一步** → 更新项目主文档"下一步行动" + 会话交接

## 每次记录至少包含什么

推荐最小结构：

```text
日期：
来源：
发生了什么：
为什么重要：
当前结论：
下一步：
关联笔记：
```

## 项目主文档字段

每个项目主文档建议保留这些段落：

- 项目目的
- 背景
- 大需求
- 小需求拆分
- 当前阶段
- 当前状态
- 关键决策
- 已完成
- 进行中
- 下一步行动
- 风险与阻塞
- 验收标准
- 相关笔记

## 开发类项目的额外要求

开发类项目必须先在 Obsidian 里完成开发前准备，再进入实现。

至少要写清楚：

- 要开发什么。
- 为什么开发。
- 第一版做到什么程度。
- 哪些功能暂时不做。
- 文件、数据、界面、流程的大致结构。
- Claude 适合做什么。
- Codex 适合做什么。
- 每一步完成后的验收标准。

实现时，AI 应按项目文档逐步执行，不应绕过项目文档直接凭聊天上下文发挥。

## AI会话结束前检查

结束前回答四个问题：

1. 这次做了什么？
2. 这次改变了什么长期状态？
3. 下次 AI 进来要先读什么？
4. 下一步最具体的动作是什么？

如果四个问题里有任何一个影响未来协作，就写进 vault。

## 默认项目记录规则

从 2026-07-05 起，用户确认以下规则作为 Codex / Claude 在本 vault 中处理项目讨论的默认记录方式。

当用户在 Codex 或 Claude 中讨论已纳入 Obsidian 控制台的项目时，如果内容涉及项目长期状态，AI 应默认写入对应项目文档，而不是只保留在聊天窗口。

不需要记录所有闲聊，只记录会影响项目后续推进的内容。

### 分流规则

```text
项目总方向、目标、边界 → 项目主文档.md
需求变化、功能拆分、暂不做 → 需求拆分.md
完成事项、当前进度、阻塞 → 任务进度.md
技术选择、取舍原因、重要判断 → 决策记录.md
资料链接、参考文档、外部证据 → 资料索引.md
下次继续、上下文交接 → 会话交接.md
跨项目的通用发现（方法论/模式/规则） → 入口文档或 AI记录规范.md
```

### 分流纠错机制

如果 AI 在本次会话中发现上一次会话在分流时写错了位置（例如应该写到决策记录的内容被写到了任务进度），应：

1. 不要删除或覆盖已有记录。
2. 在新会话的交接日志中注明"发现分流错误：XXX 内容应属于 YYY 文档"。
3. 在正确的文档中补一条，用 `> 注：本条实际发生于 YYYY-MM-DD，当时误写入 ZZZ，现补回。` 标注。
4. 如果同一类错误反复出现（≥3 次），应更新本规范的分流规则以消除歧义。

### 执行原则

- 如果用户明确说”不要记录”，则不写入 Obsidian。
- 如果内容只是闲聊、临时想法、未确认方向，可以先询问是否记录。
- 如果内容已经形成项目目标、需求、进度、决策或下一步，应默认记录。
- 写入时应简洁、结构化，避免把整段聊天原文无差别复制进去。

## 字段规范（2026-07-07 起生效）

所有文档需在顶部加 YAML frontmatter，至少包含 `type` 字段。各类型对应字段见下：

| 文档类型 | 必填字段 |
|---|---|
| 入口文档 | type, 更新时间, 当前活跃项目, 上次会话止步于 |
| 项目主文档 | type, 项目名, 路径, 定位, 当前阶段, 最近更新 |
| 需求拆分 | type, 状态, 最近更新（正文为表格） |
| 任务进度 | type, 最近更新（正文分”已完成”/”进行中或阻塞”） |
| 决策记录 | 每条含：问题/选项/选定/理由/影响范围/是否已被推翻 |
| 会话交接 | 每条含：本次输入/本次动作/当前结论/状态/下一步 |

如果用户明确说不要记录，则不写入。闲聊不自动记录。

## AI 按文档执行（需求 2.4，2026-07-12 起）

1. **开工前必读**：对应项目主文档 + 该项目最近一条会话交接（本系统读 `会话交接日志.md` 最新条）。
2. **边界**：主文档「暂不做 / 边界」与需求拆分中未开项，不得擅自开工。
3. **写回**：判断 → 决策记录或主文档；进度/阻塞 → 任务进度；需求变化 → 需求拆分；跨项目规则 → 本规范或入口。

## 开发类项目额外约束（需求 2.5）

开发前须在 Obsidian 写清：做什么、为什么、第一版程度、暂不做、文件/数据/界面结构、AI 分工、验收标准。实现按文档逐步推进，禁止一次性大规模改源码。

## 多机路径可用性（需求 2.6 / 3.5）

1. 执行涉及代码或他库路径的任务前，先核对本机路径是否存在。
2. 不可访问 → 标注「项目在其他机器，非丢失」，提示切换机器或改做纯文档任务；禁止静默跳过，禁止在未查 GitHub/备份前写「源码丢失」。
3. 项目主文档须维护「路径映射」表：当前优先路径 + 已知他机路径 + 可用性检查说明。
4. 环境变更时主动改 frontmatter/正文路径，并在交接日志写变更前后对比。

## 状态颜色标注（2026-07-13 起 · 进行中=蓝）

> **之上：** [[00_状态色标说明]]  
> **时时变源：**  
> - 文件夹：`90_系统配置/folder_status.yml`  
> - 细分文件：`90_系统配置/file_status.yml`  
> → 跑 `sync_folder_status_colors.ps1`  
> 同步范围：左侧色条 + 索引名 + **细分文件名** `<emoji>_原名.md` + wiki 链接。  
> **进行中用柔和蓝 🔵 / `[!info]`，禁止用刺眼金黄表示进行中。**

| 标记 | 含义 | 视觉 / callout | 索引文件名 | 细分文件名 |
|------|------|----------------|------------|------------|
| 🟢 | 已完成 / 稳态 | 绿边 · `[!success]` | `00_🟢索引.md` | `🟢_….md` |
| 🔵 | **进行中** | **柔和蓝边** · `[!info]` | `00_🔵索引.md` | `🔵_….md` |
| 🟠 | 日程待办 | 琥珀 · `[!abstract]` | `00_🟠索引.md` | `🟠_….md` |
| 🔴 | 阻塞 / 逾期 | 红 · `[!danger]` | `00_🔴索引.md` | `🔴_….md` |
| ⚪ | 暂停 | 灰 · 普通 info 灰调 | `00_⚪索引.md` | `⚪_….md` |

### 强制规则

1. 改文件夹忙闲：改 `folder_status.yml` → 跑 sync。  
2. 改细分文件忙闲：在 `file_status.yml` 登记（path 写**不带 emoji** 的原名）→ 跑 sync。  
3. 各区主索引必须是 `00_<emoji>索引.md`；活跃控制/跟踪笔记应有 `<emoji>_` 前缀，禁止侧栏长期无色标。  
4. 文档内未完成项必须带 🔵/🟠/🔴/⚪，禁止无色标灰字。  
5. 不要手改 AUTO 生成的 CSS，也不要手改带色标的文件名（一律走 sync）。  
6. 稳态归档（方法论/历史案例等）默认不改名；需要侧栏可见时再写入 `file_status.yml`。
