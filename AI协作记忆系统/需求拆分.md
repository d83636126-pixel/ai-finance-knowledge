# AI 协作记忆系统：需求拆分

> 本文件是 `项目_Codex与Claude协作记忆系统` 的小需求拆分。
> 每条大需求拆为可独立验收的小需求。

---

## 大需求 1：AI 长期记忆层

**目标**：AI 在新会话中能恢复关键上下文，不需要人重新解释。

### 小需求

#### 1.1 入口文档在新会话中有效

- [ ] 新 AI（Claude / Codex）读取 `00_AI记忆入口.md` 后，能准确回答：当前有几个活跃项目、各自处于什么阶段。
- [ ] 入口文档不超过一屏（避免 AI 在长文中迷失）。
- [ ] 入口文档的链接在新会话中能正常跳转。

#### 1.2 会话交接可操作

> ⚠️ 与 4.4（会话结束检查项强制执行）互为原子依赖——须同时验收，不应出现"交接内容定义好了但从未执行"的情况。

- [ ] 每次会话结束前，AI 必须更新 `会话交接日志.md`。
- [ ] 交接日志包含四项：本次做了什么、改变了什么状态、下次 AI 先读什么、下一步最具体的动作。
- [ ] 交接日志按日期倒序，最新记录在最上面。
- [ ] 下次 AI 进入时，能从交接日志反推上次的完整上下文（不需要读全部历史对话）。

#### 1.3 用户偏好可复用

- [ ] 用户的分析风格（批判性优先、跨期对照）、表达偏好（直接动手、结构化输出）已在入口文档中记录。
- [ ] 新 AI 能根据这些偏好调整自己的输出风格，不需要用户重复说明。
- [ ] 用户的 CLAUDE.md 全局指令与 vault 中的偏好记录保持一致。

#### 1.4 跨工具、跨模型一致

- [ ] 无论是 Claude、Codex 还是其他 AI 工具，阅读同一份入口文档后能得出相同的"当前状态"判断。
- [ ] 入口文档不依赖特定工具的专属语法或特性。

### 验收标准

- 在新会话中打开 vault，AI 能说出当前项目和最近一次会话做了什么。
- 用户不需要重复解释已回答过的问题。

## 大需求 2：项目控制层

**目标**：所有复杂任务通过 Obsidian 项目文档控制流程，AI 按文档执行。

### 小需求

#### 2.1 项目登记零遗漏

- [ ] 每个重要项目必须在 `AI项目控制台/00_项目总索引.md` 中登记。
- [ ] 每个项目有独立文件夹，包含主文档 + 需求拆分 + 任务进度 + 决策记录 + 会话交接。
- [ ] 项目总索引保持更新，不出现"登记了但已废弃"的僵尸项目。

#### 2.2 项目主文档字段完整

- [ ] 每个项目主文档包含：项目定位、背景、大需求、当前状态、关键决策、下一步行动、验收标准。
- [ ] 字段可以按项目类型增删，但不能缺核心字段。
- [ ] 当前状态反映的是最近一次会话后的真实状态，不是初始登记时的状态。

#### 2.3 需求拆分可执行

- [ ] 大需求拆为小需求，小需求拆为可独立验收的任务。
- [ ] 每个小需求有明确的完成标准（checkbox + 可观察的验收条件）。
- [ ] 拆分不是一次性完成，而是分阶段细化：第一版确定 MVP 边界，后续迭代前再细化下一批。

#### 2.4 AI 按文档执行

- [ ] AI 在开始工作前，必须读取项目主文档和最近的会话交接。
- [ ] AI 执行时不跳过项目文档限定的边界（暂不做的内容不擅自开始）。
- [ ] 执行过程中产生的判断、进度、阻塞，必须写回对应的子文档。

#### 2.5 开发类项目的额外约束

- [ ] 开发前必须在 Obsidian 写完：要开发什么、为什么、第一版做到什么程度、哪些暂不做、文件/数据/界面结构、AI 分工（Claude vs Codex）、验收标准。
- [ ] 实现按项目文档逐步推进，不一次性大规模修改源码。

#### 2.6 跨设备/多机协作可用性检查（2026-07-07 新增）

- [ ] AI 执行项目相关任务前，先核对当前机器与项目路径是否匹配。
- [ ] 路径不可访问时，标注"项目在其他机器，非丢失"，并提示用户切换机器或提供内容，不得静默跳过或误判为源码丢失。
- [ ] 项目主文档的 `路径` 字段使用多机映射结构（当前机器 + 已知机器列表 + 可用性检查），支持多机器协作常态。
- [ ] 优先级：P1（多机协作已确认为常态使用场景，非边缘情况）。

### 验收标准

- 关闭所有 AI 会话后，重新打开 vault，能给一个新 AI 完整的项目执行上下文。
- 不需要看聊天记录就能理解"为什么要做、做了什么、接下来怎么做"。
- 项目路径在另一台机器上不可访问时，AI 不出"源码丢失"结论，而是正确识别为多机协作常态。

## 大需求 3：迁移与同步层

**目标**：vault 可跨机器、跨工具、跨 AI 模型迁移，GitHub 是同步中枢。

### 小需求

#### 3.1 GitHub 同步可用

- [ ] vault 根目录是 Git 仓库，能推送到 GitHub。
- [ ] 配置 Obsidian Git 插件或 Git hook，实现自动或半自动推送。
- [ ] `.gitignore` 正确配置，排除 `.obsidian/workspace.json` 等个人工作区文件。

#### 3.2 克隆即用

- [ ] 在新机器上 `git clone` 后，用 Obsidian 打开 vault 即进入可用状态。
- [ ] 入口文档的 wiki 链接在新机器上能正常解析（路径名一致）。
- [ ] 不需要手动复制配置文件或调整路径。

#### 3.3 多副本一致性

- [ ] 桌面 vault（`ai-finance-knowledge`）和 D 盘 vault（`AI 金融知识点`）的内容保持一致。
- [ ] 明确"主副本"是哪个，其他副本的修改必须合并回主副本。
- [ ] 避免两个副本同时编辑同一文件导致冲突。

#### 3.4 新 AI 上手时间

- [ ] 一个新 AI（Claude / Codex / 未来其他工具）从打开入口文档到能开始执行任务，不超过 5 分钟。
- [ ] 上手时间可测量：从"读取入口文档"到"能说出当前项目和下一步行动"的轮次数。

#### 3.5 环境变更处理机制（2026-07-07 新增）

- [ ] 项目路径/机器环境发生变更时，AI 应主动更新所有受影响文档的路径字段（含 frontmatter）。
- [ ] 在会话交接中记录变更前后对比，而不是被动等待用户逐条纠正。
- [ ] 优先级：P1。

### 验收标准

- 在另一台机器 clone 仓库后，用 Obsidian 打开 vault，能看到完整的项目结构和最新状态。
- 不需要手写任何配置。

## 大需求 4：记录规范层

**目标**：AI 写入规则清晰、一致、可执行，不依赖模型临时判断。

### 小需求

#### 4.1 写入时机明确

- [ ] 当前 `AI记录规范.md` 中的"需要写入"列表是否覆盖所有实际场景。
- [ ] 增加反例：哪些情况明确不需要记录（闲聊、临时想法、已废弃的方向）。
- [ ] AI 能根据"新决定 / 新任务 / 新进度 / 新风险 / 新资料 / 下一步"六类自动判断是否需要写入。

#### 4.2 分流规则稳定

- [ ] 当前分流规则在实际使用中是否被遵守：目标→主文档、需求变化→需求拆分、进度→任务进度、决策→决策记录。
- [ ] 新增一类：跨项目的通用发现 → 入口文档或方法论。
- [ ] 分流错误要有纠正机制：如果 AI 写错了位置，下次会话应能发现并修正。

#### 4.3 命名规范统一

- [ ] 项目主文档：`项目_名称.md`。
- [ ] 需求拆分：`需求拆分.md`（每个项目固定名称）。
- [ ] 任务进度：`任务进度.md`。
- [ ] 决策记录：`决策记录.md`。
- [ ] 会话交接：`会话交接.md`。
- [ ] 资料索引：`资料_主题.md`。
- [ ] 一次性设计/实现记录：`主题_YYYY-MM-DD.md`。
- [ ] 检查当前 vault 中所有文件是否符合上述命名规范。不符合的列出并决定是否重命名。

#### 4.4 会话结束检查项强制执行

- [ ] 每次 AI 会话结束前必须回答四项：做了什么、改变了什么状态、下次先读什么、下一步最具体的动作。
- [ ] 如果四项中有任何一项影响未来协作，必须写回 vault。
- [ ] 如果没有任何影响（纯闲聊、信息查询），可以跳过记录但需明确声明。

#### 4.5 记录质量

- [ ] 写入内容结构化，不是聊天原文的无差别复制。
- [ ] 日期、来源、发生了什么、为什么重要、当前结论、下一步——六项至少覆盖四项。
- [ ] 每季度回顾一次记录质量：是否有过时信息、是否有僵尸任务、是否有写错位置的内容。

#### 4.6 决策推翻追踪（2026-07-07 新增，已生效）

- [x] 决策记录.md 中每条决策须标注"是否已被推翻"字段。
- [x] 若某决策被后续决策推翻，须回填推翻日期和原因，防止 AI 重新提出已否决方案。
- [x] 优先级：P0（已在 2026-07-07 字段规范中生效，两个子项目的决策记录已全部标注）。

#### 4.7 字段规范合规检查（2026-07-07 新增）

- [ ] 定期（建议每次重大结构调整后）扫描现有文档是否符合字段规范。
- [ ] 列出不合规文件清单并修正。
- [ ] 不要求实时检查，避免过度开销。
- [ ] 优先级：P2。

### 验收标准

- 任意打开一个项目文档，能清晰看到项目从创建到当前的全过程。
- 不会出现"这个状态是上周的，现在已经不是了"的情况。

## 大需求 5：vault 内容覆盖

**目标**：金融知识库和 AI 项目控制台双轨都纳入结构化树管理。

### 小需求

#### 5.1 金融知识库知识图谱索引更新

- [ ] `00_知识图谱索引.md` 反映当前 vault 的实际文件结构。
- [ ] 新增文件后，索引应及时更新（或明确"索引只覆盖核心文件"的规则）。

#### 5.2 金融知识库项目化

- [ ] 识别金融知识库中可以独立成"项目"的内容块（如 NFP 月度跟踪、数据博弈推演、投资复盘）。
- [ ] 为第一个金融知识库子项目建立项目主文档（建议从"NFP 月度跟踪"开始，因为它已有方法论、数据、案例、推演四类内容）。
- [ ] 子项目将已有的分散笔记用 wiki 链接串联起来，而不是移动文件。

#### 5.3 项目控制台覆盖

- [ ] `AI项目控制台/00_项目总索引.md` 始终反映所有活跃项目。
- [ ] 已废弃或暂停的项目标记 `status: inactive` 或 `status: archived`，不要直接删除。
- [ ] 每个项目的任务进度不超过一周未更新。

#### 5.4 双轨交叉引用

- [ ] 金融知识库中的方法论可以链接到项目控制台的具体实现。
- [ ] 例如 `方法论_非农注水识别手册.md` 可以链接到 `financial-alert-system` 中相关图谱节点。
- [ ] 项目控制台的设计讨论可以引用金融知识库中的数据作为证据。

### 验收标准

- vault 的 `README.md` 或 `index.md` 能引导新 AI 快速判断该读什么、不该读什么。
- 金融知识库内容不再"散落"，每条重要笔记能在项目树中找到归属。

## 暂不做

- 暂不引入第三方自动化工具（如 n8n、Zapier）自动写入 vault。
- 暂不把聊天全文自动保存到 vault（成本高、噪音大）。
- 暂不做 vault 的多人协作（当前只有一个人用）。
- 暂不做 vault 内容的全文搜索索引（Obsidian 自带搜索已够用）。

## 优先级排序

| 优先级 | 大需求 | 原因 |
|--------|--------|------|
| P0 | 1. 长期记忆层 + 4. 记录规范层 | 这两个是系统的核心价值——如果 AI 不能恢复上下文，其他都没意义 |
| P1 | 2. 项目控制层 | 已经有两个活跃子项目在用，需要确保流程稳定 |
| P2 | 3. 迁移与同步层 | GitHub 同步已有基础，主要差自动化推送；3.2（克隆即用）和 3.3（多副本一致性）已于 2026-07-07 提升为 P1 |
| P3 | 5. vault 内容覆盖 | 金融知识库的"项目化"是锦上添花，不影响系统核心功能 |
