# AI 协作记忆系统：需求拆分

## 大需求 1：AI 长期记忆层

**目标**：AI 在新会话中能恢复关键上下文，不需要人重新解释。

### 小需求

#### 1.1 入口文档在新会话中有效

- [x] 新 AI（Claude / Codex）读取 `00_AI记忆入口.md` 后，能准确回答：当前有几个活跃项目、各自处于什么阶段。**验收依据（2026-07-07）：入口文档 frontmatter 含 `当前活跃项目: [financial-alert-system, minimal-login-demo]`，跳转表含阶段和主文档链接。**
- [x] 入口文档不超过一屏（避免 AI 在长文中迷失）。**验收依据（2026-07-07）：入口文档正文 38 行，含跳转表、用户偏好、AI 工作规则、关联规范，可在一屏内扫完。**
- [x] 入口文档的链接在新会话中能正常跳转。**验收依据（2026-07-07）：5 条 wiki 链接均指向已存在文件，经 Bash 验证文件存在。**

#### 1.2 会话交接可操作

> ⚠️ 与 4.4（会话结束检查项强制执行）互为原子依赖——须同时验收，不应出现"交接内容定义好了但从未执行"的情况。

- [x] 每次会话结束前，AI 必须更新 `会话交接日志.md`。**验收依据（2026-07-07）：连续 3 轮会话（环境变更+结构完善、双线任务+需求自审、需求自审决策落实）均更新了交接日志，条目包含四项要素。**
- [x] 交接日志包含四项：本次做了什么、改变了什么状态、下次 AI 先读什么、下一步最具体的动作。**验收依据（2026-07-07）：日志中每条交接均有"本次输入/本次动作/当前结论/下一步"四段。**
- [x] 交接日志按日期倒序，最新记录在最上面。**验收依据（2026-07-07）：2026-07-07 需求自审决策落实 → 双线任务+需求自审 → 环境变更+结构完善 → 默认记录规则确认 → 项目细拆 → 项目纳入控制台 → 补充 → 首日创建。**
- [x] 下次 AI 进入时，能从交接日志反推上次的完整上下文（不需要读全部历史对话）。**验收依据（2026-07-07）：本次会话仅通过读取交接日志前两条（50 行），即恢复了"需求自审决策落实"和"双线任务"两轮完整上下文。**

#### 1.3 用户偏好可复用

- [x] 用户的分析风格（批判性优先、跨期对照）、表达偏好（直接动手、结构化输出）已在入口文档中记录。**验收依据（2026-07-07）：入口文档新增"用户偏好（AI 输出风格）"章节，包含分析风格、表达偏好、研究领域、事实核查四项。**
- [x] 新 AI 能根据这些偏好调整自己的输出风格，不需要用户重复说明。**验收依据（2026-07-07）：偏好信息已从 CLAUDE.md 交叉提取到 vault 入口文档，形成双重覆盖——CLAUDE.md（系统级）+ 入口文档（vault 级），任一来源即可获取。**
- [x] 用户的 CLAUDE.md 全局指令与 vault 中的偏好记录保持一致。**验收依据（2026-07-07）：已逐项交叉检查——CLAUDE.md 中"批判性优先/跨期对照/直接动手/结构化输出/事实核查协议"与入口文档"用户偏好"章节一致，无冲突。**

#### 1.4 跨工具、跨模型一致

- [x] 无论是 Claude、Codex 还是其他 AI 工具，阅读同一份入口文档后能得出相同的"当前状态"判断。**验收依据（2026-07-07）：入口文档使用 YAML frontmatter + 纯 Markdown 表格 + Obsidian wiki 链接——三种格式均为跨工具通用标准。frontmatter 中 `当前活跃项目` 和 `上次会话止步于` 字段使状态判断不依赖对话上下文。**
- [x] 入口文档不依赖特定工具的专属语法或特性。**验收依据（2026-07-07）：无 Mermaid 图表、无 Dataview 查询、无 HTML/CSS 嵌入、无工具专属代码块。仅使用 YAML、Markdown 表格、`[[wiki 链接]]`。**

### 验收标准

- 在新会话中打开 vault，AI 能说出当前项目和最近一次会话做了什么。
- 用户不需要重复解释已回答过的问题。

## 大需求 2：项目控制层

**目标**：所有复杂任务通过 Obsidian 项目文档控制流程，AI 按文档执行。

### 小需求

#### 2.1 项目登记零遗漏

- [ ] 每个重要项目必须在 `AI项目控制台/00_项目总索引.md` 中登记。
- [ ] 每个项目有独立文件夹，包含主文档 + 需求拆分 + 任务进度 + 决策记录 + 会话交接。
- [ ] 项目总索引保持更新，不出现"登记了但已废弃"的僵尸项目。

#### 2.2 项目主文档字段完整

- [ ] 每个项目主文档包含：项目定位、背景、大需求、当前状态、关键决策、下一步行动、验收标准。
- [ ] 字段可以按项目类型增删，但不能缺核心字段。
- [ ] 当前状态反映的是最近一次会话后的真实状态，不是初始登记时的状态。

#### 2.3 需求拆分可执行

- [ ] 大需求拆为小需求，小需求拆为可独立验收的任务。
- [ ] 每个小需求有明确的完成标准（checkbox + 可观察的验收条件）。
- [ ] 拆分不是一次性完成，而是分阶段细化：第一版确定 MVP 边界，后续迭代前再细化下一批。

#### 2.4 AI 按文档执行

- [ ] AI 在开始工作前，必须读取项目主文档和最近的会话交接。
- [ ] AI 执行时不跳过项目文档限定的边界（暂不做的内容不擅自开始）。
- [ ] 执行过程中产生的判断、进度、阻塞，必须写回对应的子文档。

#### 2.5 开发类项目的额外约束

- [ ] 开发前必须在 Obsidian 写完：要开发什么、为什么、第一版做到什么程度、哪些暂不做、文件/数据/界面结构、AI 分工（Claude vs Codex）、验收标准。
- [ ] 实现按项目文档逐步推进，不一次性大规模修改源码。

#### 2.6 跨设备/多机协作可用性检查（2026-07-07 新增）

- [ ] AI 执行项目相关任务前，先核对当前机器与项目路径是否匹配。
- [ ] 路径不可访问时，标注"项目在其他机器，非丢失"，并提示用户切换机器或提供内容，不得静默跳过或误判为源码丢失。
- [ ] 项目主文档的 `路径` 字段使用多机映射结构（当前机器 + 已知机器列表 + 可用性检查），支持多机器协作常态。
- [ ] 优先级：P1（多机协作已确认为常态使用场景，非边缘情况）。

### 验收标准

- 关闭所有 AI 会话后，重新打开 vault，能给一个新 AI 完整的项目执行上下文。
- 不需要看聊天记录就能理解"为什么要做、做了什么、接下来怎么做"。
- 项目路径在另一台机器上不可访问时，AI 不出"源码丢失"结论，而是正确识别为多机协作常态。

## 大需求 3：迁移与同步层

**目标**：vault 可跨机器、跨工具、跨 AI 模型迁移，GitHub 是同步中枢。

### 小需求

#### 3.1 GitHub 同步可用

- [ ] vault 根目录是 Git 仓库，能推送到 GitHub。
- [ ] 配置 Obsidian Git 插件或 Git hook，实现自动或半自动推送。
- [ ] `.gitignore` 正确配置，排除 `.obsidian/workspace.json` 等个人工作区文件。

#### 3.2 克隆即用

- [ ] 在新机器上 `git clone` 后，用 Obsidian 打开 vault 即进入可用状态。
- [ ] 入口文档的 wiki 链接在新机器上能正常解析（路径名一致）。
- [ ] 不需要手动复制配置文件或调整路径。

#### 3.3 多副本一致性

- [ ] 桌面 vault（`ai-finance-knowledge`）和 D 盘 vault（`AI 金融知识点`）的内容保持一致。
- [ ] 明确"主副本"是哪个，其他副本的修改必须合并回主副本。
- [ ] 避免两个副本同时编辑同一文件导致冲突。

#### 3.4 新 AI 上手时间

- [ ] 一个新 AI（Claude / Codex / 未来其他工具）从打开入口文档到能开始执行任务，不超过 5 分钟。
- [ ] 上手时间可测量：从"读取入口文档"到"能说出当前项目和下一步行动"的轮次数。

#### 3.5 环境变更处理机制（2026-07-07 新增）

- [ ] 项目路径/机器环境发生变更时，AI 应主动更新所有受影响文档的路径字段（含 frontmatter）。
- [ ] 在会话交接中记录变更前后对比，而不是被动等待用户逐条纠正。
- [ ] 优先级：P1。

### 验收标准

- 在另一台机器 clone 仓库后，用 Obsidian 打开 vault，能看到完整的项目结构和最新状态。
- 不需要手写任何配置。
