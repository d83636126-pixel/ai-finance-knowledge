# 加密微观信号 → 仓位动作 映射方法论 v1

## 0. 设计原则

1. **离散化优先**:所有原始指标先转换为有明确阈值的状态(而非连续数值),避免每次判断口径漂移。
2. **周期分层**:短线/中线/长线仓位使用不同信号池,禁止跨层污染(短线噪音不得影响核心仓位)。
3. **显式失效条件**:每个信号必须定义有效期和失效条件,过期自动降权,防止"僵尸信号"持续影响判断。
4. **矩阵先于裁量**:信号组合到仓位动作的映射优先用规则表匹配,LLM 仅在规则表未覆盖的组合上做裁量,并标注为"未校准判断"。
5. **可归因**:每次仓位建议输出必须附带触发信号列表 + 各信号权重,便于事后复盘。

## 1. 信号定义层

### 1.1 短线信号池(半衰期:小时~1天,仅影响短线/对冲仓位)

| 信号ID | 指标 | 触发条件 | 失效条件 | 有效期 |
|---|---|---|---|---|
| S-FR-01 | 资金费率(8h) | 连续3期 > 0.05% | 回落至 < 0.02% 或触发后48h | 24h |
| S-FR-02 | 资金费率(8h) | 连续3期 < -0.03% | 回升至 > -0.01% 或触发后48h | 24h |
| S-OI-01 | OI + 价格联动 | OI单日+15%且价格同向+5% | OI回落至触发前水平 | 12h |
| S-LIQ-01 | 清算热力图 | 单方向清算 > 过去30日95分位 | 清算后价格企稳2h | 6h |
| S-SKEW-01 | 25-delta skew | 单日转向绝对值 > 5% | 回归至±2%区间内 | 48h |

**待补充**:订单簿深度失衡、期权隐含波动率曲面变化——需要你确认数据源(Deribit/OKX API 可用性)后再定义阈值。

### 1.2 中线信号池(半衰期:周~月,影响战术仓位)

| 信号ID | 指标 | 触发条件 | 失效条件 | 有效期 |
|---|---|---|---|---|
| M-EXC-01 | 交易所净流入 | 单周净流入 > 90日均值+2σ | 净流入回落至均值±1σ | 2周 |
| M-EXC-02 | 交易所储备 | 创180日新低 | 储备回升5%以上 | 4周 |
| M-WH-01 | 巨鲸→交易所转账 | 单笔/单日累计 > 阈值(需按币种定义) | 无后续同向转账7天 | 2周 |
| M-WH-02 | 巨鲸→冷钱包 | 同上,方向相反 | 同上 | 2周 |

### 1.3 长线信号池(半衰期:月~季,仅影响核心仓位配置比例)

## 3. 待你决策的关键参数

在接入 evaluator 模块前,以下参数需要你根据实际风险偏好和仓位规模校准,我先列出需要你确认的项:

1. **巨鲸转账阈值**:按 BTC/ETH 分别设定绝对值(如 BTC > 500枚,ETH > 5000枚)还是按市值百分比动态调整?
2. **仓位幅度基数**:文中的百分比是相对"该周期仓位"还是相对"总仓位"?建议明确分母。
3. **信号冲突处理**:短线信号建议减仓、长线信号建议加仓时,如何处理?(建议:长线信号定调方向,短线信号仅调节执行节奏,而非反向覆盖)
4. **强信号升级规则**:多个信号池同时触发时(如短线+中线同时看空),是否需要额外的"共振加权"逻辑?

## 4. 与现有系统的接入建议

建议在 `financial-alert-system` 中新增 `position-advisor` 模块,位置在 evaluator 之后、notifier 之前:

```
scanner → on-chain → evaluator → [position-advisor] → notifier
```

`position-advisor` 输入:evaluator 输出的信号状态数组
输出结构建议:

```json
{
  "timestamp": "...",
  "triggered_signals": ["S-FR-01", "M-EXC-01"],
  "layer": "short-term",
  "action": "reduce",
  "magnitude": "-20%~-30%",
  "expiry": "2026-07-15T00:00:00Z",
  "confidence": "matrix-matched",
  "note": "若信号组合未在矩阵中定义,confidence 标注为 unmatched,交由人工判断"
}
```

这样 notifier 推送的就是结构化仓位建议而非纯文本分析,也便于你之后在 Obsidian 里做归因复盘(比对建议动作与实际价格走势)。
