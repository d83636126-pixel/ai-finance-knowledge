# 金融分析系统：把"技术+AI协作模板"落地成具体步骤

> 承接之前那份《低估监控系统六层实现指南》——那份讲的是**系统本身的技术架构**（六层流水线）。
> 这份讲的是**你怎么用AI协作模板一步步把它做出来**，并且把 harness / skills / 提示词 / loop / RAG 这几个你不熟悉的概念，一一落到这个项目里的具体位置上。

---

## 第一步：先扫盲——这几个词在你这个项目里具体是什么

不讲抽象定义，直接对应到你要做的事情上：

| 概念 | 大白话 | 在你这个项目里对应什么 |
|---|---|---|
| **Harness** | 让AI能自主干活的"运行环境/骨架程序" | 你的 `run_daily.py` 调度脚本——它负责在什么时候调用什么、传什么数据给AI、AI说完了接下来干什么。Claude Code/Cursor 本身也是一种 harness，但那是给你**开发**用的，不是给你系统**运行**用的（下面细讲这个区别） |
| **Skills** | 打包好的、可复用的"标准作业流程" | 比如"计算z-score因子"这件事你会反复做很多次，与其每次都重新描述规范，不如写成一个 Skill，让 AI 每次写这类代码时自动照着规范来 |
| **提示词 (Prompt)** | 你给AI的具体指令 | 分两部分：System Prompt（定死角色、规则、输出格式，比如"只能依据我给你的数据生成结论，禁止编造"）+ User Prompt（这次具体给它什么数据、问它什么） |
| **Loop（agentic loop）** | "做一步→检查结果→决定下一步"，反复循环，而不是一次性说完就完事 | 比如：调用数据API失败就重试3次；LLM生成解释后，再检查一遍有没有编造数字，不对就重新生成——这些"检查-决定-再来一次"的循环，就是loop |
| **RAG** | 先去找相关资料，再让AI基于这些资料回答，而不是让它凭记忆瞎编 | 你系统里"解释生成层"要做的事：先从新闻库里检索这只股票最近的相关新闻片段，再把这些片段喂给AI去生成"为什么被低估"的解释 |
| **Workflow** | 一条**固定死的**步骤顺序，每一步做什么、下一步是什么，都是你写死在代码里的，AI只负责其中某几步的具体内容，不负责决定"接下来该干嘛" | 你的 `run_daily.py` 六层流水线本身就是一个 workflow——永远是"抓数据→存储→算因子→打分→生成解释→推送"这个固定顺序，不会变 |

记住一句话就够了：**Harness 是骨架，Skills 是给骨架配的说明书，Prompt 是你跟AI说的话，Loop 是"重复检查直到满意"的机制，RAG 是"先查资料再回答"，Workflow 是"步骤顺序死不死"**。下面把这六个东西全部安进五阶段模板里。

**这里有个容易搞混但很重要的区分：Workflow vs Agent。**

- **Workflow（工作流）**：步骤顺序是你预先写死的代码逻辑决定的，LLM只在某几个固定节点被调用去完成具体子任务（比如"生成解释文字"），它不能自己决定"要不要跳过某一步"或者"接下来该做什么"。优点：行为可预测、好调试、出错好定位。
- **Agent（智能体）**：不预先写死步骤，而是让LLM自己根据情况决定下一步做什么、要不要调用哪个工具、什么时候算完成。优点：能应付你没预料到的情况；缺点：行为不可预测，调试难，容易跑偏。

**你这个金融监控系统，从头到尾都应该是个 Workflow，不是 Agent**——原因很简单：你的任务是"每天固定跑这六层"，步骤本来就是确定的，不需要AI去"决定"要不要抓数据、要不要打分。这也是为什么之前阶段1里建议"不需要上LangChain之类的agent框架，一个顺序脚本+几个函数调用就够"——因为你要的本来就是workflow，不是agent，硬上agent框架反而是给一个不需要自主决策的任务，强行加了一层不必要的不确定性。

真正需要"agent式"自主决策能力的，只有一个例外：如果你以后想让AI自己去判断"这只股票要不要额外做深度调研、调研到什么程度算够了"——那才是agent该出场的场景，属于这套系统的进阶功能，MVP阶段完全不需要。

## 第二步：五阶段模板，套到这个项目上

### 阶段0：明确目标

先把这句话写下来，贴在最显眼的地方：

> **目标**：每个交易日收盘后，自动对指定的一批医疗保健股票算出低估程度打分，触发预警时推送给我，并附带基于最近新闻的解释。
> **验收标准**：跑一整周无崩溃；抽查5只股票的分数，人工核对因子计算无误；触发的预警里，解释文字里的每一个具体数字都能在原始新闻/财报里找到出处（不是编的）。

这一步不需要AI帮你想，这是你自己要拍板的范围和标准——花10分钟写清楚，比后面反复返工省时间得多。

### 阶段1：方案讨论——这一步你要做一个关键决策：Harness 怎么搭

这是最容易被忽略但最重要的一步。你要先想清楚一件事：

**开发这个系统时用的 Claude Code/Cursor，和这个系统"跑起来"之后用的东西，是两回事。**

- Claude Code/Cursor 是你**写代码时**用来对话、生成代码的工具——一个交互式 harness，你在，它才干活。
- 但你要的是"收盘后自动跑，不用你守着"——这需要你自己写一个**无人值守的 harness**：一个 Python 脚本，被 cron 定时触发，脚本内部按顺序调用各层逻辑，其中"解释生成层"那一步会去调用 Claude 的 API（不是Claude Code，是纯API调用）。

所以这一阶段要决定的事：
1. 用什么调度：Linux服务器上直接用cron，还是先在自己电脑上手动跑着测（MVP阶段推荐后者，简单）
2. LLM怎么接入：直接用 `anthropic` 这个Python SDK 调 Messages API（前面给你的 `explain/rag.py` 骨架就是这么干的），不需要额外框架
3. 要不要一开始就上复杂的agent框架（LangChain之类）——**不需要**，你这个场景是"固定六步流水线"，不是"AI自己决定下一步做什么"的开放式任务，用简单的顺序脚本+几个函数调用就够，上重型框架反而增加你自己都看不懂的复杂度

### 阶段2：写详细计划——这一步已经帮你做完了

《低估监控系统六层实现指南》那份文件，就是这一阶段的产出物：每一层要建哪些文件、表结构长什么样、代码骨架是什么。如果还没细看，先回去过一遍，这是你阶段3要按着做的计划书。

### 阶段3：执行——这里具体展开 Skills / Prompt / Loop / RAG 怎么写

**Skills 怎么用**：如果你在 Cursor/Claude Code 里开发这套代码，写一个项目内的 Skill（比如 `.cursor/skills/finance-factor-calc/SKILL.md`），内容大概是：

```markdown
---
name: finance-factor-calc
description: 计算估值因子（z-score、DCF）时使用，确保同业比较不跨子行业混用
---
# 金融因子计算规范
- z-score 计算必须先按 sub_industry 分组，再组内比较，禁止跨子行业直接比较
- DCF 相关因子权重不超过总分的15%
- 所有因子计算函数必须同时返回原始输入值，用于事后追溯
```

以后每次让 AI 写因子相关代码，它会自动套用这份规范，不用你每次都重复讲一遍。

**Prompt 怎么写**（对应"解释生成层"）：拆成两部分，System Prompt 管死规则，User Prompt 管这次的具体数据：

```python
SYSTEM_PROMPT = """你是金融研究助手。只能依据下方提供的新闻片段和量化数据生成结论，
不能引入片段之外的事实或数字。必须输出JSON，字段：reasons, risk_flags, confidence"""

# User Prompt 每次动态拼数据进去
user_prompt = f"股票:{ticker}\n量化数据:{quant_context}\n新闻片段:{news_chunks}"
```

**Loop 怎么用**——这里给你两个具体例子，难度从低到高：

*简单的loop（不需要AI参与，纯代码）*：数据抓取失败时的重试

```python
for attempt in range(3):
    try:
        data = fetch_one(ticker, api_key)
        break
    except Exception as e:
        if attempt == 2:
            log_failure(ticker, e)  # 重试3次还失败，记录日志，跳过这只股票，不让整个批处理崩掉
```

*进阶的loop（AI参与，自我校验）*：让模型生成解释后，自己检查有没有编造

```python
def generate_with_verification(ticker, quant_context, news_chunks, max_retries=2):
    for attempt in range(max_retries + 1):
        result = generate_explanation(ticker, quant_context, news_chunks)
        # 第二次调用，专门检查第一次的输出有没有编造
        check_prompt = f"""以下是生成的分析结论：{result}
        原始新闻片段：{news_chunks}
        请检查结论里的每个具体数字/事实是否都能在原始片段中找到依据，
        只回答 VALID 或 INVALID"""
        verdict = client.messages.create(
            model="claude-sonnet-4-6", max_tokens=10,
            messages=[{"role": "user", "content": check_prompt}]
        ).content[0].text.strip()
        if verdict == "VALID":
            return result
    return {"reasons": [], "risk_flags": ["多次生成未通过校验，人工复核"], "confidence": 0.0}
```

这个"生成→自我校验→不通过就重来"的循环，就是把我们之前聊的"verification-before-completion"这个原则，直接写进代码里，而不是靠你人工每次记得检查。

**RAG 怎么用**：具体到你的 `news_chunks` 是怎么来的——不是把所有新闻都塞给模型，而是先做检索：

```python
# 用 pgvector 做相似度检索，只捞出跟"估值/业绩/监管"相关、最近30天的片段
query_embedding = get_embedding(f"{ticker} 估值 业绩 监管 新闻")
news_chunks = db.execute("""
    SELECT chunk_text FROM news_chunks
    WHERE ticker = %s AND published_at > now() - interval '30 days'
    ORDER BY embedding <-> %s LIMIT 5
""", (ticker, query_embedding))
```

先检索、再喂给模型，这就是完整的RAG——检索(Retrieval) + 增强(Augmented) + 生成(Generation)。

### 阶段4：调试——举个具体场景

假如某天某只股票的 z-score 算出来是 -47（明显异常），系统性调试的做法：
1. 列假设：可能是①这只股票当天数据缺失/异常值 ②历史窗口数据本身有问题 ③除以了一个接近0的标准差
2. 一次只查一个：先打印这只股票当天的原始输入值，看是不是数据源本身返回了异常数字
3. 定位到根因后再修（比如加一层"标准差小于阈值时跳过z-score计算"的保护），而不是直接把这个异常值滤掉了事——不然下次同样的数据问题还会再犯

### 阶段5：验证——具体验证清单

- [ ] 连续跑7天不崩溃（哪怕某天某只股票数据缺失也不影响其他55只）
- [ ] 抽查5只股票的z-score，手工用Excel验算一遍，跟脚本结果对得上
- [ ] 抽查3条触发的预警，把解释文字里提到的每个具体数字，回去原始新闻里核对，看有没有编造
- [ ] 故意断网/给一个错误的API key，确认脚本能优雅报错而不是直接卡死或者发出一个内容为空的错误预警

不要想着一次性把 harness、skills、RAG 全部配置好再开始——按这个顺序，每一步都是能跑起来看到结果的最小单元：

4. **再之后**：解释生成层（RAG+Prompt+Loop）和推送层，这时候再回头看这份指南的阶段3部分

## 第四步：还有什么值得加、该在什么时候加

**核心原则：不要一次性全部搭好。** 跟你之前认同的"架构要有规划，但不代表要过度设计"是一回事——没被验证过有没有用的技术，提前搭进去，大概率是在给一个还没跑起来的系统增加不必要的复杂度，而且会拖慢你验证核心逻辑的速度。

下面这些都是有用的技术，但排序是"该在什么信号出现时再加"，不是"越早加越好"：

| 技术 | 解决什么问题 | 什么时候加 |
|---|---|---|
| **结构化输出校验** | 防止 LLM 返回的 JSON 格式不对，导致下游代码解析时直接崩溃 | 阶段3一开始就该有——这不算"进阶技术"，是基本卫生，前面 `rag.py` 骨架里的 try/except JSONDecodeError 就是这个 |
| **基础日志/可观测性** | 出问题时能查到"当时到底发给了模型什么、模型返回了什么"，不然调试全靠猜 | MVP刚跑起来、遇到第一次诡异结果时就该加，不用等很久——建议从第一天起就把每次LLM调用的输入输出都存一份 |
| **熔断/异常检测** | 万一某天数据源整体挂了、返回一堆垃圾数据，系统不该照常推送一堆基于垃圾数据算出来的错误预警 | 数据源+计算层稳定运行之后就该加，属于"能不能放心让它自动跑"的门槛，别等真出了一次乌龙预警才想起来加 |
| **回测框架** | 在真的开始信任这套打分系统之前，先看历史上如果一直用这套规则，信号准不准 | 系统积累至少一个季度的历史打分数据之后，正式依赖它做决策之前，这是必须做的一步 |
| **Prompt 回归测试** | 你改了一下提示词，怎么确定这次改动是让解释质量变好了还是变差了，而不是凭感觉 | 开始频繁调提示词、又说不清哪次改动效果变好变差的时候加 |
| **成本监控** | LLM API调用是要花钱的，量上来了容易失控，尤其是接了自我校验的loop之后（一次生成变成两三次调用） | 一旦从"手动测试"切换成"真的每天自动跑"，就该加，别等账单出来才后知后觉 |
| **数据版本/血缘追溯** | 能回答"为什么上周三这只股票算出来是这个分数"——审计用的 | 只有当你需要跟别人解释/审计这套系统的判断依据时才需要，个人自用的MVP阶段可以先不做 |
| **多模型交叉验证** | 一个模型说被低估，换个角度/换个模型问会不会有不同结论，增加确信度 | 系统已经稳定运行一段时间、你想进一步提高预警可信度时的进阶做法，早期没必要，也没跑够数据支撑这种投入 |

**分几批加，不要一口气全上：**

- **第一批（写代码的时候就该有，不算可选）**：结构化输出校验、基础的重试/日志
- **第二批（系统稳定跑1-2周之后）**：更完整的可观测性、熔断/异常检测
- **第三批（攒够至少一个季度历史数据之后）**：回测框架、Prompt回归测试
- **第四批（真打算长期认真维护、或者要给别人用的时候）**：成本监控看板、数据版本血缘、多模型交叉验证

判断"现在该不该加某个技术"最简单的问题就是：**这个技术解决的是我已经实际遇到的问题，还是我想象中未来可能会遇到的问题？** 前者现在加，后者先记在待办里，等它真的发生了再加。

## 第五步：业务上怎么落地——四阶段信任建立框架

前面几步讲的是"怎么把系统造出来"（技术架构 + AI协作流程），这一步讲的是"造出来之后，业务上怎么一步步敢用它、用到多大程度"——这是两条不同的线，很多人容易忽略后者，系统一跑通就直接拿真金白银去信它，这是最容易出事的地方。

这个框架不是金融专用的，本质上是任何"AI辅助决策系统"从能跑到真正被信任都要走的路，跟软件工程里的灰度发布、金融风控里的"挑战者模型"验证思路是同一个逻辑。

### 阶段A：纸上验证 / 影子模式（Shadow Mode）

- **做什么**：系统正常跑，正常产出预警和打分，但你完全不基于它做任何真实操作——只是记录"如果当时信了它，结果会怎样"
- **目的**：在不承担任何真实风险的情况下，先看清楚这套系统的真实准确率、误报率是多少
- **通过标准**：连续跑够长时间（比如一个季度以上），手动核对了足够多的预警（比如50条以上），对它的"脾气"有了直觉判断——知道它在什么情况下容易误判（比如财报季前后波动大时噪音会变多）

### 阶段B：小规模试点

- **做什么**：拿一笔完全能承受损失的小额资金，针对系统触发的预警做实际操作，同时记录"这次是完全照系统做的，还是加了自己的判断"
- **目的**：验证从"纸面上准"到"真金白银操作时准"之间有没有落差——比如数据延迟、执行时机偏差、操作时的心理因素导致偏离系统建议
- **通过标准**：这个阶段的核心不是赚钱，而是验证整条执行链路没问题——数据没延迟、预警送达够及时、你实际能照着系统的节奏操作

### 阶段C：信任校准

- **做什么**：回头看阶段A/B积累的数据，针对性调整权重、阈值、质量过滤器——前面提到的回测框架，这时候才真正派上用场
- **目的**：这不是"调一次就完事"，而是建立一个"回头看→调整→再观察"的循环
- **通过标准**：调整后的版本，在没被用来调参的那部分历史数据上（样本外数据）表现依然稳定——如果只在用来调参的那部分数据上表现好，很可能是过拟合，调出来的是"刚好拟合过去"而不是真的有效

### 阶段D：扩大范围 + 持续监控

- **做什么**：股票池扩大、参考权重提高，同时建立固定的复盘节奏
- **目的**：市场环境、行业政策会变，原来成立的假设可能不再成立，系统需要被持续检验，而不是搭好了就一劳永逸
- **通过标准**：有固定的复盘节奏（比如每季度一次），每次都重新问一遍"这套系统的前提假设是不是还站得住"

### 贯穿全程都要守住的一条线

不管走到哪个阶段，**系统的输出永远只是决策辅助，不是替代决策**——这条线不应该随着你对系统越来越信任而放松。恰恰相反，系统自动化程度越高、你越依赖它，一旦某个假设悄悄失效而你没发现，影响范围可能越大，人工复核这道门槛反而更该留着，不是等信任建立起来了就可以撤掉。

### 一个具体的例子，把四阶段串起来看

假设系统在某个交易日对 UNH 算出综合分数 -2.3（触发 strong 级预警），解释生成层给出"远期P/E低于5年历史均值2个标准差、净负债/EBITDA在合理范围"——四个阶段分别怎么处理这一条：

- **阶段A**：只记录这条预警，不操作。两周后回头看，UNH股价有没有按预期修复，人工标记这条预警"事后看是否准确"，存进跟踪表。
- **阶段B**：如果阶段A积累的记录显示这类预警historically比较可靠，才用小额资金试着操作一次，同时记录"这次买入价和系统预警时的价格差了多少"（执行延迟造成的偏差）。
- **阶段C**：攒够几十条记录后，回头看是不是 managed_care 这个子行业的权重配置(relative_valuation 0.45)偏高或偏低，调整后一定要在没用来调参的那部分历史数据上再验证一遍。
- **阶段D**：每季度复盘时问一句"UNH所在的管理式医疗保险这个子行业，估值锚点(远期P/E)还合适吗"——比如遇到医保报销政策大改的年份，估值逻辑可能整个要重新评估。

### 配套跟踪表格

写代码维护这些记录不如直接用表格——我做了一份可以直接开始用的 Excel 跟踪表，四个阶段各一个 sheet，已经放了示例行、下拉框（预警等级、是否遵照系统执行等）和自动统计公式（比如阶段A的"是否达到进入阶段B的门槛"会自动算），不用你自己再搭。
